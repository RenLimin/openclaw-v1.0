## 🏗️ 架构总览

### 🎯 设计原则

| 原则 | 说明 |
|------|------|
| **统一接入** | 所有数据源（离线/在线/记忆）走同一处理管道 |
| **结构化优先** | 尽可能转化为结构化数据存储，而非纯向量 |
| **可追溯性** | 每条知识都有源数据、处理时间、版本号 |
| **增量更新** | 支持按源路径增量更新，无需全量重建 |
| **独立查询** | 知识库可独立于智能体运行，提供 API 服务 |
| **自进化** | 使用反馈自动优化索引、权重、检索算法 |

### 📦 技术栈选型

| 层级 | 技术选型 | 选型理由 |
|------|---------|---------|
| **主数据库** | SQLite 3.x | 轻量、零依赖、单文件、嵌入式、Python 原生支持 |

| **OCR识别** | 百炼视觉模型 / PaddleOCR | 中文识别准确率高 |

### 📂 数据源一：离线文档（本地/网络路径）

#### 🎯 支持格式

| 分类 | 支持格式 | 优先级 |
|------|---------|--------|
| 📄 文档类 | `.md`, `.txt`, `.docx`, `.doc`, `.pdf`, `.rtf` | P0 |
| 📊 表格类 | `.xlsx`, `.xls`, `.csv`, `.json`, `.yaml`, `.yml` | P0 |
| 📐 演示类 | `.pptx`, `.ppt` | P1 |
| 🌐 网页类 | `.html`, `.htm`, `.mhtml` | P1 |
| 🖼️ 图片类 | `.png`, `.jpg`, `.jpeg`, `.tiff`, `.bmp` | P1 (需OCR) |
| 📧 邮件类 | `.eml`, `.msg` | P2 |
| 📦 压缩包 | `.zip`, `.rar`, `.7z` | P2 (需先解压) |

#### 🔧 处理流程

```
输入：本地路径 / 网络路径 (SMB/NFS/AFP)
    ↓
1️⃣ 路径扫描与验证
   - 验证路径可访问性
   - 递归扫描所有文件
   - 按扩展名过滤支持格式
   - 记录文件元数据（大小、修改时间、MD5）
    ↓
2️⃣ 文档解析
   - 按格式调用对应解析器
   - 提取文本内容 + 元数据
   - 表格类保留结构化信息
   - 图片类触发 OCR
    ↓
3️⃣ 文本分块
   - 语义分块（按标题/段落）
   - 固定大小分块（默认 512 tokens）
   - 保留分块上下文（页码、标题路径）
    ↓
4️⃣ 向量化（可选）
   - 调用 Embedding 模型
   - 生成向量表示
    ↓
5️⃣ 存储入库
   - 写入 SQLite 主库
   - 建立全文索引
   - 建立向量索引（如启用）
```

#### ⚙️ 配置示例

```yaml
data_sources:
  offline_docs:
    - name: 产品文档库
      path: /Volumes/Documents/产品文档/
      scan_interval: 24h  # 自动扫描更新间隔
      recursive: true
      includes: ["*.md", "*.docx", "*.pdf"]
      excludes: ["*tmp*", "*draft*"]
      ocr_enabled: true  # 对图片类启用 OCR
      
    - name: 合同模板库
      path: //server/contract_templates/
      credentials:
        username: kb_reader
        password: "${KB_PASSWORD}"  # 环境变量注入
```

### 🌐 数据源二：在线文档 / Web 页面

#### 🎯 支持类型

| 类型 | 支持场景 | 认证方式 |
|------|---------|---------|
| 公开网页 | 公开可访问 URL | 无 |
| 企业 Wiki | Confluence / Notion / 飞书 / ONES | Token / 账号密码 |
| 文档平台 | 语雀 / 石墨 / Google Docs | API Key / OAuth |
| OA 系统 | 泛微 / 致远 | Cookie / 账号密码 / SSO |
| 代码仓库 | GitHub / GitLab / Gitee | Personal Access Token |

#### 🔧 处理流程

```
输入：URL + 认证信息（可选）
    ↓
1️⃣ 认证与访问
   - 静态页面：直接 HTTP 请求
   - 动态页面：Playwright 渲染
   - 需认证：注入 Cookie / Token / 模拟登录
    ↓
2️⃣ 内容提取
   - 文章正文提取（排除导航/广告/侧边栏）
   - 链接提取（用于发现相关页面）
   - 元数据提取（标题、作者、时间）
    ↓
3️⃣ 增量更新检测
   - 记录页面 ETag / Last-Modified
   - 记录内容哈希值
   - 仅提取有变化的页面
    ↓
4️⃣ 去重与清洗
   - 移除重复内容
   - 清洗 HTML 噪音
   - 标准化格式
    ↓
5️⃣ 分块与入库
   - 同离线文档流程
   - 保留 URL 作为溯源信息
```

#### ⚙️ 配置示例

```yaml
data_sources:
  web_pages:
    - name: ONES 产品 Wiki
      base_url: https://ones.bangcle.com/wiki/
      auth_type: cookie
      cookie_file: ~/.openclaw/cookies/ones.json
      crawl_depth: 2
      include_patterns: ["/product/", "/docs/"]
      exclude_patterns: ["/login", "/logout"]
      update_interval: 12h
      
    - name: 企业公开知识库
      base_url: https://kb.bangcle.com/
      auth_type: none
      crawl_depth: 3
      follow_links: true
```

5️⃣ 关联与入库
   - 与已有知识做关联分析

| 特性 | 说明 |
|------|------|
| **零依赖** | Python 原生支持，无需额外安装数据库服务 |

### 📋 核心表结构设计

#### 1. `knowledge_sources` - 数据源注册表

```sql
CREATE TABLE knowledge_sources (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL UNIQUE,           -- 数据源名称
    type TEXT NOT NULL,                   -- offline_doc / web_page / agent_memory
    config_json TEXT NOT NULL,            -- 完整配置 JSON
    status TEXT DEFAULT 'active',         -- active / inactive / error
    last_scan_at TIMESTAMP,               -- 上次扫描时间
    next_scan_at TIMESTAMP,               -- 下次计划扫描
    total_documents INTEGER DEFAULT 0,    -- 总文档数
    total_chunks INTEGER DEFAULT 0,       -- 总分块数
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

#### 2. `documents` - 文档主表

```sql
CREATE TABLE documents (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    source_id INTEGER NOT NULL,           -- 关联数据源
    uri TEXT NOT NULL,                    -- 唯一资源标识（路径/URL）
    title TEXT,                           -- 文档标题
    doc_type TEXT NOT NULL,               -- 文档类型（扩展名/MIME）
    file_size INTEGER,                    -- 文件大小（字节）
    content_hash TEXT NOT NULL,           -- MD5/SHA256 内容哈希（用于增量检测）
    last_modified_at TIMESTAMP,           -- 源文件修改时间
    metadata_json TEXT,                   -- 扩展元数据
    chunk_count INTEGER DEFAULT 0,        -- 分块数量
    token_count INTEGER DEFAULT 0,        -- 总 Token 数
    status TEXT DEFAULT 'active',         -- active / archived / deleted
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    FOREIGN KEY (source_id) REFERENCES knowledge_sources(id),
    UNIQUE(source_id, uri)                -- 同一数据源内 URI 唯一
);

-- 索引
CREATE INDEX idx_documents_source ON documents(source_id);
CREATE INDEX idx_documents_type ON documents(doc_type);
CREATE INDEX idx_documents_hash ON documents(content_hash);
```

#### 3. `document_chunks` - 文档分块表

```sql
CREATE TABLE document_chunks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    document_id INTEGER NOT NULL,         -- 关联文档
    chunk_index INTEGER NOT NULL,         -- 分块序号（从0开始）
    chunk_title TEXT,                     -- 分块所属标题
    content TEXT NOT NULL,                -- 分块文本内容
    content_length INTEGER NOT NULL,      -- 文本长度
    token_count INTEGER,                  -- Token 估算
    page_number INTEGER,                  -- 页码（如适用）
    context_path TEXT,                    -- 上下文路径（标题层级）
    metadata_json TEXT,                   -- 分块级元数据
    
    FOREIGN KEY (document_id) REFERENCES documents(id),
    UNIQUE(document_id, chunk_index)
);

-- 全文索引（FTS5）
CREATE VIRTUAL TABLE chunk_fts USING fts5(
    content,
    chunk_title,
    context_path,
    tokenize = 'unicode61 remove_diacritics 0'
);
```

#### 4. `chunk_vectors` - 向量索引表（可选）

```sql
CREATE TABLE chunk_vectors (
    chunk_id INTEGER PRIMARY KEY,
    vector BLOB NOT NULL,                 -- 向量二进制数据（FAISS 格式）
    model_name TEXT NOT NULL,             -- Embedding 模型名称
    model_version TEXT,                   -- 模型版本
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    FOREIGN KEY (chunk_id) REFERENCES document_chunks(id)
);
```

#### 5. `knowledge_tags` - 知识标签体系

```sql
CREATE TABLE knowledge_tags (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL UNIQUE,            -- 标签名
    category TEXT,                        -- 分类（领域/优先级/类型）
    color TEXT DEFAULT '#808080',         -- 展示颜色
    description TEXT,                     -- 标签描述
    usage_count INTEGER DEFAULT 0,        -- 使用次数
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE document_tags (
    document_id INTEGER NOT NULL,
    tag_id INTEGER NOT NULL,
    confidence REAL DEFAULT 1.0,          -- 标签置信度
    tagged_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    PRIMARY KEY (document_id, tag_id),
    FOREIGN KEY (document_id) REFERENCES documents(id),
    FOREIGN KEY (tag_id) REFERENCES knowledge_tags(id)
);
```

#### 6. `knowledge_relations` - 知识关联关系

```sql
CREATE TABLE knowledge_relations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    from_doc_id INTEGER NOT NULL,
    to_doc_id INTEGER NOT NULL,
    relation_type TEXT NOT NULL,          -- references / duplicate / related / supersedes
    strength REAL DEFAULT 0.5,            -- 关联强度 0-1
    description TEXT,                     -- 关系描述
    discovered_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    FOREIGN KEY (from_doc_id) REFERENCES documents(id),
    FOREIGN KEY (to_doc_id) REFERENCES documents(id),
    UNIQUE(from_doc_id, to_doc_id, relation_type)
);
```

#### 7. `query_logs` - 查询日志（用于优化）

```sql
CREATE TABLE query_logs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    query_text TEXT NOT NULL,             -- 查询文本
    top_results TEXT,                     -- TOP 结果 ID 列表
    clicked_result INTEGER,               -- 用户点击的结果（如可收集）
    feedback_score INTEGER,               -1/0/+1 用户反馈评分
    execution_time_ms INTEGER,            -- 执行耗时
    result_count INTEGER,                 -- 返回结果数
    queried_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    agent_name TEXT                       -- 哪个智能体发起的查询
);
```

### 📡 API 接口定义

#### 1. 基础查询

```python
from knowledge_base import KnowledgeBase

kb = KnowledgeBase("~/.openclaw/knowledge-base/knowledge.db")

# 简单关键词查询
results = kb.search("合同违约责任", limit=10)

# 语义查询（需 Embedding 模型）
results = kb.semantic_search("如果对方违约了我该怎么办?", limit=10)

# 混合查询（关键词 + 语义 + 过滤）
results = kb.hybrid_search(
    query="违约金",
    filters={
        "doc_type": ["pdf", "docx"],
        "source_id": [1, 3],  # 限定数据源
        "tags": ["合同模板", "法务审核"]
    },
    limit=20
)
```

## 🔄 增量更新机制

### 🎯 设计目标

- ✅ **增量检测**：只更新有变化的文件，不重复处理
- ✅ **自动调度**：按配置的时间间隔自动扫描更新
- ✅ **版本保留**：保留历史版本，支持回滚
- ✅ **冲突处理**：源文件冲突时的处理策略

---

### 📅 增量更新流程

```
触发：定时任务 / 手动触发 / 文件系统事件
    ↓
1️⃣ 源扫描
   - 遍历数据源路径/URL
   - 收集所有文件元信息（路径、大小、修改时间、哈希）
    ↓
2️⃣ 变更检测（与数据库中记录对比）
   ├─ 新增：哈希不存在 → 全新处理 ✓
   ├─ 更新：哈希不同 + 修改时间更新 → 删除旧分块，重新处理 ✓
   └─ 不变：哈希相同 → 跳过 ✗
    ↓
3️⃣ 批量处理
   - 多线程并行处理变更文件
   - 进度实时记录
    ↓
4️⃣ 索引更新
   - 更新全文索引
   - 更新向量索引
    ↓
5️⃣ 完成通知
   - 统计：新增X个、更新Y个、跳过Z个
   - 记录更新日志
```

### ⚙️ 更新策略配置

```yaml
update_strategies:
  # 策略1：定时扫描（适合变化不频繁的文档库）
  - type: scheduled
    cron: "0 2 * * *"  # 每天凌晨2点
    sources: ["产品文档库", "合同模板库"]
    
  # 策略2：文件系统监听（适合本地高频变更目录）
  - type: filesystem_watch
    path: /Volumes/Documents/实时更新/
    debounce: 300  # 300秒防抖，批量更新时避免重复触发
    sources: ["实时协作文档"]
    
  # 策略3：手动触发
  - type: manual
    description: "需要管理员手动触发更新"
```

## 📊 知识质量评估体系

### 🎯 质量维度

| 维度 | 评估指标 | 权重 |
|------|---------|------|
| **完整性** | 内容是否完整、有无缺失页码/章节 | 25% |
| **准确性** | 内容是否准确、有无错误信息 | 25% |
| **时效性** | 内容是否最新、是否已过期 | 20% |
| **结构化程度** | 是否可转化为结构化数据 | 15% |
| **可复用性** | 是否可被多个场景引用 | 15% |

---

### 📈 质量评分流程

```
新文档入库
    ↓
自动评分：
  ✓ OCR 识别准确率估算
  ✓ 文本完整性检测（断页、乱码）
  ✓ 时效性检测（创建/修改时间）
  ✓ 结构化程度（表格、列表占比）
    ↓
人工审核（低分文档触发）：
  分数 < 60分 → 推送人工审核队列
    ↓
审核反馈 → 优化解析算法 → 重新处理
```
