## 🏗️ MCP 四层控制架构（通用）

**Model Control Plane - 拦住 80% 不必要的大模型调用**

```
用户请求
    ↓
┌─────────────────────────────────────────────────┐
│  1. 路由层：简单任务脚本搞定                       │
│     目标：脚本解决率 ≥ 70%                        │
│     逻辑：关键词匹配 → 规则引擎 → 脚本执行        │
├─────────────────────────────────────────────────┤
│  2. 缓存层：相同问题直接返回                       │
│     目标：缓存命中率 ≥ 50%                        │
│     实现：SQLite + LRU 淘汰策略                  │
├─────────────────────────────────────────────────┤
│  3. 熔断层：错误/超限时自动降级                    │
│     目标：可用性 ≥ 99.5%                         │
│     策略：连续 5 次错误 → 降级到纯脚本模式        │
├─────────────────────────────────────────────────┤
│  4. 预算层：控制调用量，防止成本失控               │
│     目标：月度成本偏差 < 10%                      │
│     告警：用到 50% 预警 → 80% 告警 → 95% 限流   │
└─────────────────────────────────────────────────┘
```

### 通用成本公式

```
大模型实际成本 = 理论成本 × 脚本解决率 × (1 - 缓存命中率)
```

**P1 优化目标：让成本降为原来的 15%！**

### 🔹 Git分支策略（GitFlow简化版，行业最佳实践）

```
main 分支（受保护）
  ↓ 只能从release分支合并
  ↓ 每一次合并都打Tag
release/vX.X.X 分支
  ↓ 测试通过后合并到main
develop 分支
  ↓ 日常开发分支
feature/xxx 分支
  ↓ 功能开发分支
hotfix/xxx 分支
  ↓ 紧急修复分支
```

#### 分支保护规则（3条红线，不可违反）
| 分支 | 保护规则 |
|------|---------|
| **main** | ❌ 禁止直接push！只能通过PR合并 |
| **release** | ❌ 禁止直接push！只能从develop合并 |
| **develop** | ✅ 可以push，但必须通过CI检查 |

## 🔐 新增：敏感数据加解密提交GitHub机制（安全红线）

### 核心理念
**密码、Token、密钥、私钥绝对不能明文提交到GitHub！无论公有库还是私有库！**

---

### 🔹 3层防护体系（行业标准）

#### 第1层：.gitignore 前置防护（必须配置！）
```
# 敏感文件（永远不要提交！）
.env
.env.local
.env.*.local
.secrets
*.key
*.pem
*.p12
*.pfx
id_rsa*

# 配置文件模板（可以提交，但占位！）
.env.example
config.example.json
```
**强制要求**：所有项目必须有.gitignore，且包含以上内容！

---

#### 第2层：Git Hook 预检防护（pre-commit）
```bash
#!/bin/bash
# pre-commit hook：提交前自动检查敏感数据

# 检查是否有密码、Token、密钥
SECRET_PATTERNS=(
  'password='
  'token='
  'api_key='
  'secret_key='
  'private_key'
  '-----BEGIN RSA'
  '-----BEGIN PRIVATE'
)

for pattern in "${SECRET_PATTERNS[@]}"; do
  if git diff --cached | grep -q "$pattern"; then
    echo "❌ 检测到敏感数据模式: $pattern"
    echo "请删除敏感数据后再提交！"
    exit 1
  fi
done

echo "✅ 敏感数据预检通过"
exit 0
```
**强制要求**：所有Agent的工作环境必须配置pre-commit hook！

---

#### 第3层：配置文件加密存储（git-crypt / SOPS）

##### 方案A：git-crypt（轻量透明加密，推荐）
```
# 初始化（首次）
git-crypt init

# 生成密钥
git-crypt export-key ../.git-crypt-key

# 配置加密规则（.gitattributes）
secret.env filter=git-crypt diff=git-crypt
config/secrets.json filter=git-crypt diff=git-crypt
*.key filter=git-crypt diff=git-crypt

# 正常提交，自动透明加密
git add secret.env
git commit -m "chore: 添加加密配置文件"

# 解密查看
git-crypt unlock
```

##### 方案B：Mozilla SOPS（功能强大，支持多KMS）
```
# 加密配置文件
sops -e config/secrets.yaml > config/secrets.enc.yaml

# 解密编辑
sops config/secrets.enc.yaml

# 运行时自动解密
export SOPS_AGE_KEY=xxx
sops exec-env secrets.enc.yaml 'python app.py'
```

### 🔹 敏感数据管理最佳实践（5条军规）

| # | 规则 | 违反后果 |
|---|------|---------|
| 1 | **永远不要明文提交任何密码、Token、密钥** | 仓库泄露，全公司安全事故 |
| 2 | **永远不要硬编码敏感数据在代码中** | 代码一泄露，全部完蛋 |
| 3 | **敏感数据只能通过环境变量注入** | 配置与代码彻底分离 |
| 4 | **密钥文件只能存储在本地，永不提交** | 密钥泄露 = 系统被黑 |
| 5 | **定期轮换所有密钥，至少90天一次** | 长期不换，风险累积 |

---

### 🔹 敏感数据泄露应急响应流程（发生时立即执行）

```
第1步：立即撤销泄露的密钥/Token（越快越好！）
    ↓
第2步：立即强制回滚Git仓库，删除泄露的提交历史
    ↓
第3步：全面审计日志，检查是否有未授权访问
    ↓
第4步：通知所有相关人员，评估影响范围
    ↓
第5步：根因分析，制定预防措施，更新检查清单
    ↓
第6步：全员安全培训，确保同类错误永不复发
```
**响应时间要求**：敏感数据泄露必须在**15分钟内**启动应急响应！
