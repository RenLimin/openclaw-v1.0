# 🧠 人类反馈机制 (Human-in-the-Loop) v1.0

> **版本**: v1.0
> **发布日期**: 2026-05-09
> **核心目标**: 关键节点必须人工确认，确保系统可控可干预

---

## 🎯 5 类必须人工介入的场景（强制执行）

| 场景ID | 场景名称 | 触发条件 | 介入级别 |
|--------|---------|---------|---------|
| HITL-001 | **Git 提交操作** | 任何 `git commit` / `git push` | 🔴 必须确认 |
| HITL-002 | **生产数据修改** | 数据库写操作、文件删除、配置覆盖 | 🔴 必须确认 |
| HITL-003 | **大成本调用** | 单次预估 Token > $1.0 或 10万 Token | 🟠 强烈建议确认 |
| HITL-004 | **架构重大变更** | 核心架构重构、废弃已有机制 | 🟠 强烈建议确认 |
| HITL-005 | **不可撤回操作** | 发布上线、权限变更、对外通知 | 🔴 必须确认 |

## 🔄 标准三阶段流程

```
┌─────────────────────────────────────────────────────────────┐
│                     HITL 标准流程                             │
├─────────────────────────────────────────────────────────────┤
│                                                               │
│   Phase 1: PAUSE  暂停                                        │
│   ─────────────────────────────────────────────────────      │
│   • 立即暂停当前操作流程                                       │
│   • 收集操作上下文信息（影响范围、风险评估）                   │
│   • 生成「人工确认请求」卡片                                  │
│   • 启动超时计时器（默认 30 分钟）                           │
│                                                               │
│                      ↓ 用户输入 ↓                             │
│                                                               │
│   Phase 2: DECISION  决策                                    │
│   ─────────────────────────────────────────────────────      │
│   ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    │
│   │  ✅ 确认执行 │    │  🛑 终止操作 │    │  🔄 修改后执行 │    │
│   └─────────────┘    └─────────────┘    └─────────────┘    │
│                                                               │
│                      ↓ 继续流程 ↓                             │
│                                                               │
│   Phase 3: RESUME  恢复                                       │
│   ─────────────────────────────────────────────────────      │
│   • 根据决策执行对应分支                                      │
│   • 记录完整决策链路（谁、什么时间、什么决策）               │
│   • 决策结果自动沉淀到知识库                                  │
│                                                               │
└─────────────────────────────────────────────────────────────┘
```

## 🔙 回滚机制设计

### 回滚触发条件
1. 用户明确选择「终止操作」
2. 超时未响应（默认 30 分钟）
3. 执行过程中发生严重错误

### 回滚执行步骤
```
Step 1: 记录回滚触发原因与时间
Step 2: 执行逆向操作（如已执行部分操作）
Step 3: 恢复到暂停前的精确状态
Step 4: 生成完整回滚报告
Step 5: 记录到错误模式库（ERRORS.md）
```

## 💾 反馈自动沉淀知识库机制

### 决策记录标准字段
```json
{
  "hitl_id": "HITL-001-20260509-142035",
  "scenario_type": "git_commit",
  "trigger_time": "2026-05-09T14:20:35+08:00",
  "decision_time": "2026-05-09T14:22:18+08:00",
  "decision": "approved", // approved/rejected/modified
  "context_summary": "提交 v4.2.0 架构演进规划",
  "risk_assessment": "低风险，文档更新",
  "user_feedback": "确认提交",
  "outcome": "success",
  "lessons_learned": "无"
}
```

### 沉淀知识库位置
- **短期（1个月）**: `memory/hitl_records/YYYY-MM/`
- **中期（1年）**: `knowledge-base/hitl_decisions/` SQLite 表
- **长期（永久）**: 提炼为最佳实践，晋升到 `AGENT.md` 规则
