# 🔗 调用链追踪框架 v1.0

> **版本**: v1.0
> **发布日期**: 2026-05-09
> **核心目标**: 复杂请求全链路可追溯，瓶颈可定位，根因可分析
> **设计标准**: 完全兼容 OpenTelemetry / Zipkin / Jaeger 协议

---

## 🎯 解决的核心问题

### 痛点场景
```
❌ 问题1：一个任务花了10分钟，不知道时间都花在哪了
❌ 问题2：某个环节慢了，不知道是Agent卡了、工具卡了、还是大模型卡了
❌ 问题3：跨多个Agent协作的任务，出问题不知道是谁的责任
❌ 问题4：性能优化没有数据支撑，不知道该优化哪个环节
❌ 问题5：发生错误后，不知道之前的调用上下文，无法复现
```

## 🔌 5 类标准 Span 类型

| 类型ID | 类型名称 | 触发场景 | 采集内容 |
|--------|---------|---------|---------|
| S1 | Agent 生命周期 Span | Agent启动/停止/重启 | 版本、配置、启动参数 |
| S2 | 任务执行 Span | 任务开始/结束/失败 | 任务类型、输入输出、耗时 |
| S3 | 大模型调用 Span | 每次LLM推理 | 模型、Token数、耗时、参数 |
| S4 | 工具调用 Span | 每次工具调用 | 工具名、参数、返回、耗时 |
| S5 | 黑板操作 Span | 黑板读写操作 | 层级、Key、Value大小 |

### 2. 瓶颈自动定位算法

```python
def find_bottleneck(trace):
    """自动定位链路瓶颈"""
    spans = trace['spans']

    # 规则1: 耗时Top3的Span
    top_3_by_duration = sorted(spans, key=lambda x: x['duration_ms'], reverse=True)[:3]

    # 规则2: 耗时占总时间 > 30% 的Span
    total_duration = trace['duration_ms']
    high_ratio_spans = [s for s in spans if s['duration_ms']/total_duration > 0.3]

    # 规则3: 调用频率 > 10次的外部调用
    high_freq_spans = [s for s in spans if s['span_kind'] == 'CLIENT' and ...]

    return {
        "top_bottlenecks": top_3_by_duration,
        "high_ratio_spans": high_ratio_spans,
        "recommendations": generate_optimization_recommendations(top_3_by_duration)
    }
```

### 3. 错误根因自动分析

```python
def analyze_error_root_cause(trace):
    """自动分析错误根因"""
    failed_spans = [s for s in trace['spans'] if s['status'] == 'failed']

    for span in failed_spans:
        # 检查是否是根源（父Span成功，子Span失败）
        parent_span = find_parent_span(span['parent_span_id'])
        if parent_span and parent_span['status'] == 'success':
            return {
                "root_cause_span": span,
                "error_type": span['error_info']['type'],
                "error_message": span['error_info']['message'],
                "impact_scope": calculate_impact_scope(span, trace),
                "retry_possible": check_retry_possible(span)
            }
```

## 📈 链路聚合统计与趋势分析

### 按维度聚合的标准报表

| 聚合维度 | 统计指标 | 报表用途 |
|---------|---------|---------|
| 按任务类型聚合 | 平均耗时、成功率、错误率 | 发现哪类任务最慢/最容易出错 |
| 按Agent类型聚合 | 平均耗时、Token消耗、错误率 | Agent能力评估与优化 |
| 按模型类型聚合 | 平均耗时、Token消耗、成本 | 成本优化与模型选型 |
| 按工具类型聚合 | 平均耗时、成功率、重试率 | 工具稳定性评估 |
| 按时间段聚合 | 小时/天/周维度的趋势 | 性能趋势预测与容量规划 |

### 性能基线与告警

```python
# 建立各维度的性能基线
baseline = {
    "task_type_a": {"p50": 5000, "p95": 15000, "p99": 30000},
    "task_type_b": {"p50": 12000, "p95": 30000, "p99": 60000},
    "model_gpt4o": {"p50": 2000, "p95": 5000, "p99": 10000}
}

# 告警规则：超过基线P99的2倍即告警
def check_performance_anomaly(span):
    span_type = get_span_type(span)
    baseline_p99 = baseline[span_type]['p99']
    if span['duration_ms'] > 2 * baseline_p99:
        trigger_alert(
            alert_type="PERFORMANCE_ANOMALY",
            message=f"{span['span_name']} 耗时{span['duration_ms']}ms 超过基线2倍"
        )
```

## 🔍 典型问题场景分析示例

### 场景1：任务整体超时失败

```
问题现象：任务执行了30分钟超时失败
链路分析步骤：
1. 查看Trace总览，找到最后一个成功的Span
2. 查看失败Span的错误信息与调用栈
3. 分析失败Span的父Span链，找到第一个失败点
4. 检查失败点前后的事件（events）列表
5. 查看当时的系统资源指标（CPU/内存/网络）

输出：根因分析报告 + 修复建议 + 预防措施
```

### 场景2：任务执行缓慢但最终成功

```
问题现象：任务花了平时2倍的时间但最终成功
链路分析步骤：
1. 生成耗时瀑布图
2. 自动定位耗时Top3的Span
3. 对比历史基线，看是哪个环节变慢了
4. 分析该环节变慢的原因（Token太多？模型问题？工具慢？）
5. 给出针对性优化建议

输出：性能分析报告 + 优化建议 + 预期收益
```

### 场景3：错误率突然升高

```
问题现象：10分钟内错误率从1%突然升到20%
链路分析步骤：
1. 聚合所有错误Span，按错误类型分组
2. 按Agent/工具/模型维度交叉分析
3. 检查是单点问题还是全局问题
4. 回溯问题发生的时间点，对应有什么变更
5. 确认是否是第三方服务故障（API/网络/工具）

输出：故障根因报告 + 影响范围 + 恢复措施
```

## 🔧 存储与查询设计

### 存储策略

| 数据类型 | 热存储时长 | 冷存储时长 | 存储介质 |
|---------|-----------|-----------|---------|
| 原始Trace数据 | 7天 | 30天 | Elasticsearch |
| Span聚合指标 | 30天 | 1年 | Prometheus + TSDB |
| 趋势分析报表 | 30天 | 永久 | MySQL |
| 错误根因分析报告 | 30天 | 永久 | MySQL |

### 标准查询API

```sql
-- 1. 根据TraceID查完整链路
SELECT * FROM traces WHERE trace_id = 'TRACE-7A3F9C2E-541D-4B8C-9E7D-12345678';

-- 2. 查询所有失败的Trace
SELECT * FROM traces WHERE status = 'failed' AND start_time > NOW() - INTERVAL 1 HOUR;

-- 3. 查询耗时TOP10的Trace
SELECT * FROM traces ORDER BY duration_ms DESC LIMIT 10;

-- 4. 按Agent聚合错误率
SELECT agent_id,
       COUNT(*) as total,
       SUM(CASE WHEN status = 'failed' THEN 1 ELSE 0 END) as failed,
       SUM(CASE WHEN status = 'failed' THEN 1 ELSE 0 END) * 100.0 / COUNT(*) as error_rate
FROM spans
WHERE start_time > NOW() - INTERVAL 24 HOUR
GROUP BY agent_id
ORDER BY error_rate DESC;
```
