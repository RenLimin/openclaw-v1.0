## 🔴 0 级强制规则：工作目录限制

> 🚨 **所有技能必须 100% 遵守，无例外**

### 0.1 技能运行目录限制

**所有技能的读取、写入、执行操作仅限在 `~/.openclaw` 目录（即 `/Users/bangcle/.openclaw`）内进行，除非 Rex 单独发送明确指令授权。**

### 0.2 技能输出目录要求

| 技能输出类型 | 强制输出目录 | 说明 |
|------------|------------|------|
| **日志文件** | `~/.openclaw/logs/{skill-name}/` | 所有日志必须写入此目录 |
| **临时文件** | `~/.openclaw/tmp/{skill-name}/` | 执行完成后自动清理 |
| **数据文件** | `~/.openclaw/workspace/data/{skill-name}/` | 持久化数据存储 |
| **报告输出** | `~/.openclaw/workspace/training-reports/` | 训练报告、分析报告等 |
| **配置文件** | `~/.openclaw/workspace/skills/{skill-name}/config/` | 技能专属配置 |

### 0.3 违规处理

- 任何技能违反此规则，Jerry 有权立即终止其执行
- 输出到非法目录的文件必须立即迁移或删除
- 多次违规的技能需要重新审核和训练

## 1. 技能定义

技能是智能体团队的 **核心颗粒度**，必须满足：

- **完整性**：包含执行任务所需的全部代码、配置、文档
- **独立性**：不依赖其他技能（共享工具除外，见第 2 节）
- **可验证**：有明确的安装验证步骤
- **可回滚**：有版本号，可回退到任意历史版本

---

## 2. 技能分类

| 分类 | 定义 | 存储位置 | 示例 |
|------|------|----------|------|
| **域技能** | 只属于一个 agent，不被其他 agent 依赖 | `agents/{name}/workspace/skills/` | `contract-approval`、`ones-api` |
| **共享工具** | 通用能力，所有 agent 可引用 | `workspace/skills/` | `feishu-doc`、`web-pilot` |
| **外部技能** | 从 ClawHub/GitHub 安装 | `workspace/skills/` 或 `agents/*/workspace/skills/` | `tavily-search`、`clawhub` |
| **本地技能** | 外部技能经修改后固化为本地的技能 | 同域技能或共享工具 | 定制版 `github`、`web-pilot` |

---

## 3. 技能包结构

每个技能必须是一个完整的目录，包含以下文件：

```
skills/{skill-name}/
├── SKILL.md                # [必需] 技能说明（使用方法、触发条件）
├── VERSION                 # [必需] 当前版本号（语义化版本: X.Y.Z）
├── _meta.json              # [必需] 技能元数据（见第 4 节）
├── DEPENDENCIES.md         # [必需] 依赖清单（见第 5 节）
├── CHANGELOG.md            # [推荐] 版本变更日志
├── README.md               # [推荐] 详细说明
├── scripts/                # [可选] 辅助脚本
├── references/             # [可选] 参考资料
└── main.py / main.js       # [可选] 主执行文件
```

## 6. 技能生命周期

```
安装/创建 → 验证 → 使用 → 更新检查 → (更新/保持)
                                      ↓
                              已本地修改？ → 是 → 利弊分析 → Rex 审批 → 更新/保持
                                      ↓ 否
                              直接更新 → 验证 → 完成
```

### 6.1 技能训练完成后的独立更新

当某个 agent 的技能完成训练或阶段性训练后，**agent 可自行发起 GitHub 提交**：

```
技能训练完成 / 阶段性训练完成
    │
    ▼
1. 更新 training-memory.md
2. 更新技能 VERSION（如有变更）
3. 更新 _meta.json
4. 确保 DEPENDENCIES.md 完整
5. git add + commit + push（仅自己的文件）
6. 向 Jerry 汇报提交结果
```

**权限**：agent 只能提交自己目录下的文件（见 `github-rules.md` 第 6 节）

## 7. 定期更新检查机制

### 7.1 检查频率

| 技能类型 | 检查频率 | 检查方式 | 执行者 |
|----------|----------|----------|--------|
| 外部技能（ClawHub/GitHub） | **每周** | `clawhub` CLI / `gh` API | Jerry |
| 本地技能（已修改的外部技能） | **每周** | 检查上游新版本 | Jerry |
| 域技能（纯本地开发） | 不需要 | N/A | — |
| 共享工具（纯本地开发） | 不需要 | N/A | — |

### 7.2 更新检查流程（外部技能）

```
每周六 09:30（Cron 任务）
    │
    ▼
Jerry 遍历所有外部技能
    │
    ├── 有新版本？
    │   ├── 否 → 记录"已是最新"，结束
    │   │
    │   └── 是 → 检查本地是否有修改
    │       │
    │       ├── 无修改 → 直接更新 → 验证 → 提交 Git
    │       │
    │       └── 有修改 → 生成利弊分析报告 → 等待 Rex 审批
```

### 7.3 更新检查流程（本地技能/已修改的外部技能）

当技能已被本地修改时，**不自动更新**，执行以下流程：

1. **检查上游新版本**：记录新版本号和变更内容
2. **生成利弊分析报告**：

```markdown
# 技能更新评估报告

## 基本信息
- 技能: {skill-name}
- 当前版本: X.Y.Z（已本地修改）
- 上游新版本: X.Y+1.Z
- 检查日期: YYYY-MM-DD

## 上游变更内容
- {列出主要变更点}

## 更新利弊分析

### ✅ 更新收益
- {新功能/修复/性能提升}

### ❌ 更新风险
- {可能丢失本地修改}
- {兼容性问题}
- {需要重新训练/验证}

### 📊 本地修改对比
| 本地修改项 | 上游是否已包含 | 更新后是否冲突 |
|------------|----------------|----------------|
| {修改1} | 否 | 是/否 |
| {修改2} | 是 | 否 |

## 建议
- [ ] 建议更新（本地修改已合并到上游）
- [ ] 建议暂不更新（本地修改重要且上游未包含）
- [ ] 建议更新但需要手动合并

## 审批
- [ ] Rex 审批通过
- [ ] Rex 驳回
- [ ] Rex 要求进一步分析
```

3. **等待 Rex 审批**：
   - 审批通过 → 执行更新 → 手动合并本地修改 → 验证 → 提交
   - 驳回 → 记录原因，下次检查日再评估
   - 要求进一步分析 → Jerry 补充信息后重新提交

## 8. 技能安装/卸载规则

### 8.1 安装
- 从 ClawHub 安装: `clawhub install {skill-name}`
- 从 GitHub 安装: 克隆到对应 skills 目录
- 本地创建: 按第 3 节结构创建
- 安装后必须: 验证功能、更新 `_meta.json`、提交 Git

### 8.2 卸载
- 先确认没有其他技能依赖该技能
- 备份当前版本到 `skills/_archive/{skill-name}/`
- 从 `_meta.json` 标记 `uninstalledAt`
- 提交 Git

## 9. 版本管理

### 9.1 语义化版本 (SemVer)
- `MAJOR.MINOR.PATCH`（如 2.1.0）
- MAJOR: 不兼容的 API 变更
- MINOR: 向后兼容的功能新增
- PATCH: 向后兼容的问题修复

### 9.2 版本更新规则
- 修复 bug → PATCH +1
- 新增功能 → MINOR +1
- 不兼容变更 → MAJOR +1
- 每次版本变更必须更新 CHANGELOG.md

## 11. L0 层核心技能：业务目标深度分析标准流程

> 🚨 **强制规则**：创建任何智能体团队之前，必须先完成业务目标深度分析并获得用户确认

### 11.1 分析触发条件

自动触发业务目标分析的场景：
- 用户提出「创建/搭建/设计」智能体团队
- 用户描述了具体业务场景需要自动化
- 现有智能体团队需要重大调整或重构
- 用户提出的需求不明确，需要进一步拆解

### 11.2 标准三阶段分析流程

#### 阶段一：目标解构（必须完成）

```
输入：用户自然语言描述的业务目标
    ↓
1. 核心目标识别（5个问题必须回答）
   - 要解决的核心问题是什么？
   - 谁是最终受益者/用户？
   - 成功的具体衡量标准是什么？
   - 有什么时间限制/里程碑要求？
   - 有什么预算/资源约束？
    ↓
2. 利益相关者映射（影响力/兴趣矩阵）
   - 关键决策者：高影响力、高兴趣 → 重点沟通
   - 核心用户：高影响力、低兴趣 → 满足需求
   - 监督者：低影响力、高兴趣 → 保持知情
   - 受影响者：低影响力、低兴趣 → 适当考虑
    ↓
3. 风险与约束识别（10项通用风险检查）
   - 合规风险、成本风险、时间风险、技术风险
   - 数据风险、人员风险、变更风险、安全风险
   - 集成风险、培训风险
    ↓
输出：结构化的目标定义文档
```

#### 阶段二：能力映射（必须完成）

```
输入：阶段一的目标解构结果
    ↓
1. 核心能力需求矩阵（三级优先级）
   - 🔴 必须有（Must-have）：没有就无法实现目标
   - 🟡 应该有（Should-have）：强烈建议，大幅提升效果
   - 🟢 可以有（Nice-to-have）：锦上添花
    ↓
2. 角色推导与职责划分（遵循4原则）
   - 单一职责原则：每个角色只负责一类能力
   - 高内聚低耦合：角色内部能力相关，角色间减少依赖
   - 能力均衡：避免某个角色过重或过轻
   - 协调者唯一：有且只有一个总协调者（Jerry）
    ↓
3. 协作流程设计
   - 典型任务从哪里开始？
   - 需要经过哪些处理步骤？
   - 每个步骤由哪个角色负责？
   - 步骤之间需要传递什么信息？
   - 质量检查点在哪里？
   - 异常情况如何处理？
   - 最终交付物是什么？
    ↓
输出：完整的智能体团队需求蓝图
```

#### 阶段三：方案输出（必须完成）

```
输入：前两阶段的所有分析结果
    ↓
1. 团队架构蓝图（三层架构）
   - 协调层：Jerry（全局协调）
   - 业务层：根据能力需求推导的专业角色
   - 基础设施层：Nova + Orion + Luna（标准配置）
    ↓
2. 成功指标体系（三层KPI）
   - L1 业务结果指标：用户满意度、ROI、效率提升等
   - L2 团队效能指标：响应时间、准确率、协作成功率等
   - L3 运行健康指标：可用性、错误率、自进化频率等
    ↓
3. 四阶段实施路线图
   - 种子期（0-7天）：MVP验证
   - 生长期（7-30天）：能力完善
   - 成熟期（30-90天）：性能优化+自进化
   - 卓越期（90天+）：持续进化
    ↓
4. ROI预估与投资回报分析
   - 成本预估：开发+运行+维护
   - 收益预估：人力+效率+质量
   - 投资回收期、年化ROI计算
    ↓
输出：7份标准交付物（见11.3）
```

### 11.3 标准交付物清单（7份，缺一不可）

| 编号 | 交付物名称 | 格式 | 验收标准 |
|------|-----------|------|---------|
| 1 | 业务目标分析报告 | Markdown | 问题清晰、目标明确、所有利益相关者覆盖 |
| 2 | 团队架构蓝图 | YAML | 三层架构完整、角色职责清晰 |
| 3 | 角色定义集 | YAML | 遵循CrewAI风格，包含背景、目标、技能、工具、权限 |
| 4 | 成功指标体系 | Excel/Markdown | 三层KPI完整，每个指标有基线+目标+测量方法 |
| 5 | 实施路线图 | Markdown | 四阶段划分清晰，里程碑明确，交付物具体 |
| 6 | ROI分析报告 | Markdown | 成本收益数据具体，投资回收期计算合理 |
| 7 | 风险评估与应对 | Markdown | 10项通用风险全覆盖，每项有应对措施 |

### 11.4 用户确认流程（强制执行）

```
完成全部分析并生成7份交付物
    ↓
向用户展示完整方案
    ↓
用户提出修改意见
    ↓
根据意见迭代修改（可多轮）
    ↓
用户明确表示「确认」「同意」「可以开始」
    ↓
✅ 记录确认信息（时间、用户、确认内容）
    ↓
进入智能体团队创建流程
```

⚠️ **严禁跳过确认直接创建团队**

### 11.5 自动化工具使用规范

使用 `goal_analyzer.py` 进行自动化分析：

```bash
# 基础用法
python goal_analyzer.py --goal "用户的业务目标描述"

# 从文件读取（适合长文本）
python goal_analyzer.py --file business_goal.txt

# 指定输出目录
python goal_analyzer.py --goal "..." --output ./analysis-result/
```

自动化工具输出结果后，Jerry必须：
1. 审核自动生成的内容是否准确
2. 根据具体业务场景补充细节
3. 补充行业特定的风险和约束
4. 确保所有7份交付物完整
5. 整理成用户易读的格式提交确认

### 11.6 质量验收标准

一份合格的业务目标分析必须满足：

| 检查项 | 验收标准 | 权重 |
|--------|---------|------|
| 目标清晰度 | 核心目标能用一句话说清楚，没有歧义 | 20% |
| 可衡量性 | 所有成功指标都是可量化、可测量的 | 20% |
| 完整性 | 覆盖了所有核心能力需求，没有遗漏 | 15% |
| 合理性 | 目标在给定的时间和资源约束下是可行的 | 15% |
| 一致性 | 角色职责没有重叠，也没有遗漏 | 15% |
| 风险识别 | 识别了所有主要风险，并有应对措施 | 10% |
| 用户确认 | 用户明确签字确认 | 5% |
| **总分** | **100%** | |

- **≥ 90分**：优秀，可直接进入创建流程
- **≥ 70分**：良好，需要补充细节后确认
- **< 70分**：不合格，需要重新分析
