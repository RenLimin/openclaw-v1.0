# 火山引擎 Coding Plan 违规风险深度分析 v1.0

> ⚠️ **背景**: Volcengine 是我们当前的默认 Provider，使用专用 Coding API 端点
> 端点：`https://ark.cn-beijing.volces.com/api/coding/v3`

---

## 一、已知信息与推断

### 1.1 关键配置信息

| 项 | 值 | 风险提示 |
|----|-----|---------|
| **端点** | `/api/coding/v3` | ✅ **Coding Plan 专用端点**，不是通用推理端点 |
| **模型** | `ark-code-latest` | 专用 Coding 模型 |
| **服务商** | 字节跳动/火山引擎 | 风控严格，与抖音/头条同体系 |

### 1.2 行业通用 Coding Plan 规则（类比阿里云）

所有大厂的 Coding Plan 本质都是 **"编程助手补贴套餐"**，规则高度相似：

| 规则 | 阿里云百炼 | 火山引擎（推断） |
|------|-----------|-----------------|
| **仅限编程工具使用** | ✅ Claude Code、OpenClaw 等 | ✅ 相同（OpenClaw 在名单内） |
| **禁止 API 自动化** | ✅ 禁止自动化脚本、批量调用 | ✅ **大概率相同** |
| **禁止非交互式调用** | ✅ 禁止无人值守调用 | ✅ **大概率相同** |
| **禁止作为后端服务** | ✅ 禁止 | ✅ **大概率相同** |
| **账号专享** | ✅ 禁止共享 | ✅ 相同 |

### 1.3 字节跳动风控特点（基于公开信息）

| 特点 | 说明 | 风险等级 |
|------|------|---------|
| **异常行为检测极强** | 字节拥有全中国最成熟的反作弊系统（抖音/头条/游戏） | 🔴 高 |
| **行为模式分析** | 不只是看 QPS，更看调用模式是否像人类 | 🔴 高 |
| **零容忍机制** | 一旦判定违规直接封号，申诉难度大 | 🔴 高 |
| **关联风控** | 一个账号违规，可能牵连同主体下其他账号 | 🟡 中 |

## 二、我们当前哪些行为涉嫌违规？

### 🟡 【中风险】需要警惕的行为

#### 1. Cron 定时任务（已删除的邮件巡检）
```
风险：如果以后重新启用 Cron 并默认用 Volcengine
├── 触发方式：定时自动触发，无用户交互
├── 调用模式：非交互式
└── 结论：⚠️ 中等风险 - Coding Plan 通常禁止无人值守
```

#### 2. 子 Agent 自动调用链
```
Jerry 自动分派 → Ella/Oliver 自动执行
├── 触发方式：用户发起，但中间过程无确认
├── 调用模式：半自动化、链式调用
└── 结论：⚠️ 边界模糊 - 取决于"是否是编程场景"
```

#### 3. 批量技能训练/评估
```
批量生成训练数据、批量测试
├── 触发方式：可能是用户发起，但执行过程全自动
├── 调用模式：批量、模板化
└── 结论：⚠️ 中高风险 - "批量调用"是敏感词
```

#### 4. 深夜高频调用
```
22:00 之后仍在频繁调用
├── 行为特征：不符合正常程序员作息
├── 风控信号：疑似机器人/自动化
└── 结论：⚠️ 中风险 - 触发行为异常检测
```

### ✅ 【低风险】明确允许的行为

#### 5. 用户直接交互的编程对话
```
用户: "帮我写个 Python 脚本"
Jerry: "好的，这是脚本..."
├── 触发方式：用户主动发起
├── 场景：编程相关
├── 模式：交互式
└── 结论：✅ 完全允许 - Coding Plan 设计初衷
```

#### 6. 代码调试、解释、重构
```
用户: "这个 bug 怎么修？"
用户: "解释一下这段代码"
├── 场景：纯编程辅助
└── 结论：✅ 完全允许
```

## 四、火山引擎 Coding Plan 使用红线

> 🚨 **虽然火山引擎规则不透明，但宁可信其有，不可信其无！**
> 按照比阿里云更严格的标准执行！

### 🔴 【绝对禁止】高压线 - 碰了必冻

| 禁令 | 说明 |
|------|------|
| ❌ 禁止纯自动化 Cron 调用 | 定时任务无人值守 = 典型违规 |
| ❌ 禁止批量数据处理 | 批量生成、批量评估 = "非交互式批量调用" |
| ❌ 禁止作为服务后端 | 任何形式的产品化后端调用 = 违规 |
| ❌ 禁止 API Key 共享 | 套餐是个人版，共享 = 直接封号 |
| ❌ 禁止绕过内容安全 | 任何形式的 prompt injection 绕过 = 零容忍 |

### 🟡 【严格限制】需要控制的行为

| 限制项 | 阈值 | 说明 |
|--------|------|------|
| **单日最大调用** | ≤ 200 次 | 个人开发者正常用量不会超过这个数 |
| **单任务连续调用** | ≤ 10 次 | 一个任务链调用不超过 10 次 |
| **调用时间窗口** | 7:00 - 24:00 | 尽量符合正常人类作息，避开凌晨 1-6 点 |
| **调用间隔** | ≥ 1s 随机延迟 | 避免毫秒级连续调用 |
| **QPS 限制** | ≤ 2 次/秒 | 个人使用不会超过这个频率 |

### ✅ 【完全允许】鼓励的使用场景

| 场景 | 说明 |
|------|------|
| ✅ 用户主动发起的编程对话 | "帮我写代码"、"帮我 debug" |
| ✅ 代码解释、重构、优化 | 纯编程辅助场景 |
| ✅ 技术问题解答 | 编程相关的技术问题 |
| ✅ 脚本编写与调试 | 自动化脚本开发辅助 |
| ✅ 架构设计讨论 | 技术方案讨论 |

## 五、Provider 风险分级与使用策略

### 5.1 最终风险矩阵

| Provider | 风险等级 | 最佳使用场景 | 禁止场景 |
|----------|---------|-------------|---------|
| **DeepSeek** | 🟢 低风险 | 自动化任务、Cron、批量处理、非工作时间 | 无（通用 API） |
| **Volcengine** | 🟡 中风险 | 用户交互式编程对话、代码辅助 | 自动化、Cron、批量、凌晨 |
| **Ali** | 🟡 中风险 | 通用对话、非编程场景 | 自动化、Cron |
| **bailian** | 🔴 高风险 | 仅限纯代码任务，且用户明确指定 | 其他所有场景 |

### 5.2 智能路由策略

```
新任务到达
   │
   ├── 检测：是否为 Cron/定时触发？
   │   └── 是 → 强制使用 DeepSeek
   │
   ├── 检测：是否为批量处理任务？
   │   └── 是 → 强制使用 DeepSeek
   │
   ├── 检测：当前时间是否在 1:00-7:00？
   │   └── 是 → 强制使用 DeepSeek
```

## 六、调用行为优化（模拟人类程序员）

> 🧠 **核心思路**：让调用模式看起来像一个真实的程序员在使用编程助手，而不是机器人

| 优化项 | 机械模式（易被检测） | 人类模式（安全） |
|--------|---------------------|-----------------|
| **调用间隔** | 0ms，立即连续调用 | 1-5s 随机延迟，模拟思考时间 |
| **调用分布** | 均匀分布，24 小时 | 集中在工作时间，有明显高峰低谷 |
| **连续调用** | 一口气 20+ 次 | 每 5-10 次后有较长停顿（30s+） |
| **输入长度** | 固定模板，高度相似 | 多样化，有人类语言特征 |
| **输出长度** | 稳定、可预测 | 波动大，有时短有时长 |
| **错误重试** | 立即重试，指数退避 | 重试间隔有随机性，不机械 |

## 七、熔断与降级机制

### 7.1 Volcengine 三级熔断

| 级别 | 触发条件 | 动作 |
|------|---------|------|
| **一级预警** | 单日调用 > 150 次 | 日志告警，准备切换 |
| **二级限流** | 单日调用 > 200 次 | 自动切换到 DeepSeek |
| **三级熔断** | 连续 3 次 4xx 错误 | 熔断 1 小时，切换 DeepSeek |

### 7.2 应急预案

```
检测到 Volcengine 账号被封
   │
   ▼
1. 立即停止所有 Volcengine 调用
   │
   ▼
2. 自动全局切换到 DeepSeek 作为默认
   │
   ▼
3. 检查调用日志，分析可能的违规点
   │
   ▼
4. 提交工单申诉（准备好使用场景说明）
   │
   ▼
5. 复盘整改，避免其他 Provider 重蹈覆辙
```

## 八、总结：核心原则

### 原则 1：DeepSeek 是自动化任务的唯一选择
- 所有 Cron、定时、批量、无人值守任务 → 只能用 DeepSeek

### 原则 2：Volcengine 只用于人类交互式编程场景
- 用户："帮我写代码" → ✅ 用 Volcengine
- 系统："自动执行 XX 任务" → ❌ 绝不能用 Volcengine

### 原则 3：永远模拟人类行为模式
- 有思考间隔、有作息规律、有错误重试的随机性
- 绝不连续高频调用，绝不 24 小时不间断

### 原则 4：不把鸡蛋放一个篮子里
- 保持至少 2 个可用 Provider（现在是 Volcengine + DeepSeek）
- 任何时候都能快速切换
