# 🎰 状态机工作流模式 v1.0

> **版本**: v1.0
> **发布日期**: 2026-05-09
> **核心能力**: 复杂任务流程可控、可观测、可回滚、可断点续跑
> **设计灵感**: 借鉴 UML 状态机 + AWS Step Functions + 工作流引擎最佳实践

---

## 🎯 解决的核心问题

### 痛点场景
```
❌ 问题1：复杂任务执行到一半失败，不知道从哪里继续，只能重来，浪费大量时间
❌ 问题2：任务进度黑盒，不知道当前执行到哪一步，还有多久完成
❌ 问题3：异常发生后无回滚机制，系统处于不一致状态，人工恢复成本高
❌ 问题4：无法暂停任务人工干预后再继续，执行过程"一旦启动只能跑完"
❌ 问题5：无法预测任务时间和资源需求，任务管理盲跑
```

## 📋 7 种标准状态定义

| 状态ID | 状态名 | 英文名 | 说明 | 允许的后续状态 |
|--------|--------|-------|------|---------------|
| S0 | 待执行 | PENDING | 任务已定义，未开始 | RUNNING, CANCELLED |
| S1 | 执行中 | RUNNING | 正在执行，心跳更新中 | SUCCESS, FAILED, PAUSED, CANCELLED |
| S2 | 已暂停 | PAUSED | 人工暂停，保留完整上下文 | RUNNING, CANCELLED |
| S3 | 已成功 | SUCCESS | 正常执行完成 | 无（最终状态） |
| S4 | 失败 | FAILED | 执行出错，等待处理 | ROLLBACK, RETRY, CANCELLED |
| S5 | 回滚中 | ROLLBACK | 执行回滚补偿逻辑 | CANCELLED |
| S6 | 已取消 | CANCELLED | 任务已终止，不产生影响 | 无（最终状态） |

## 🔧 核心概念定义

### 1. 步骤 (Step) - 工作流的最小单元

```json
{
  "step_id": "S001",
  "step_name": "解析合同文件",
  "description": "读取并解析上传的合同PDF文件",
  "timeout_seconds": 300, // 5分钟超时
  "max_retries": 3, // 最大重试次数
  "retry_strategy": "exponential_backoff", // 退避策略: fixed/exponential/immediate
  "continue_on_failure": false, // 失败后是否继续后续步骤
  "requires_approval": false, // 执行前是否需要HITL人工确认
  "compensation_action": "delete_parsed_files", // 失败后的补偿动作ID
  "input_schema": {
    "file_path": "string",
    "file_type": "pdf|docx"
  },
  "output_schema": {
    "contract_id": "string",
    "parsed_data": "object"
  }
}
```

## 🔌 9 个标准 API 接口

| API | 功能 | 说明 |
|-----|------|------|
| `define(steps, transitions)` | 定义工作流 | 注册所有步骤和状态转换规则 |
| `start(workflow_id, inputs)` | 启动执行 | 从 PENDING → RUNNING，开始执行第一步 |
| `pause(workflow_id, reason)` | 暂停执行 | RUNNING → PAUSED，保留完整上下文 |
| `resume(workflow_id)` | 恢复执行 | PAUSED → RUNNING，从断点继续 |
| `retry(workflow_id, step_id)` | 重试步骤 | FAILED → RUNNING，从失败步骤重试 |
| `cancel(workflow_id, reason)` | 取消执行 | 任何状态 → CANCELLED |
| `rollback(workflow_id)` | 执行回滚 | FAILED → ROLLBACK → CANCELLED |
| `get_status(workflow_id)` | 查询状态 | 返回当前状态、进度、已耗时间 |
| `get_snapshot(workflow_id)` | 获取快照 | 返回完整执行上下文，用于持久化 |

## 🧩 4 种重试退避策略

| 策略ID | 策略名称 | 算法说明 | 适用场景 |
|--------|---------|---------|---------|
| S1 | 立即重试 | 失败后立刻重试 | 瞬时网络抖动、偶发故障 |
| S2 | 固定间隔 | 每次等待固定秒数 | 已知恢复时间的外部依赖 |
| S3 | 线性退避 | 每次等待时间线性增加 | 资源限流类故障 |
| S4 | 指数退避 | 每次等待时间指数级增长 | 绝大多数分布式场景（默认） |

```python
# 指数退避示例
retry_wait = base_seconds * (2 ** retry_count) + jitter
# 第1次：2秒，第2次：4秒，第3次：8秒，第4次：16秒...
```

## 🔄 补偿机制与 Saga 模式

### 每个步骤必须定义正向操作 + 逆向补偿操作

```
Step 1: 创建订单
  正向操作：INSERT INTO orders ...
  补偿操作：DELETE FROM orders WHERE id = ?
    ↓
Step 2: 扣减库存
  正向操作：UPDATE inventory SET count = count - ?
  补偿操作：UPDATE inventory SET count = count + ?
    ↓
Step 3: 扣款支付
  正向操作：CALL payment_api(...)
  补偿操作：CALL refund_api(...)
    ↓
✅ 全部成功 → 提交完成
❌ Step3 失败 → 执行 Step2 补偿 → 执行 Step1 补偿 → 最终状态一致
```

## 💾 断点续跑与持久化

### 状态持久化标准流程

```
事件触发（每步完成/状态变更/心跳）
    ↓
生成执行上下文快照
    ↓
写入 SQLite 工作流表
    ↓
关联 Blackboard 黑板条目
    ↓
可选：写入 Redis 缓存加速查询
```

### 崩溃恢复流程

```
系统重启后扫描工作流表
    ↓
发现状态为 RUNNING 的任务（崩溃前未正常结束）
    ↓
检查最后心跳时间
    ↓
距离最后心跳 < 超时时间 → 标记为 PAUSED，等待人工确认
距离最后心跳 > 超时时间 → 标记为 FAILED，触发回滚
    ↓
人工确认后决定：RETRY / CANCEL
```
