# 🖥️ Blackboard 共享黑板模式 v1.0

> **版本**: v1.0
> **发布日期**: 2026-05-09
> **核心目标**: 多 Agent 高效共享中间状态，解决协作中的信息孤岛问题
> **设计灵感**: 借鉴 AI 领域经典的 Blackboard Architecture Pattern（黑板模式）

---

## 🎯 解决的核心问题

### 痛点场景
```
❌ 问题1：Agent A 完成的计算结果，Agent B 不知道，重复计算，浪费资源
❌ 问题2：多个 Agent 并行工作，状态分散，需要人工汇总，效率低
❌ 问题3：任务中途需要换 Agent 接手，上下文丢失，一切重来
❌ 问题4：无法发现不同 Agent 产出之间的关联与矛盾
❌ 问题5：无全局视图，无法做全局优化与决策
```

## 🏗️ 核心架构设计

### 三层黑板结构

```
┌─────────────────────────────────────────────────────────────┐
│                    🖥️ 共享黑板 (Blackboard)                  │
├─────────────────────────────────────────────────────────────┤
│                                                               │
│  L3: 决策层 - 最终结论、全局决策、汇总结果                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 最终交付物 │ 里程碑状态 │ 全局决策 │ 风险汇总         │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                               │
│  L2: 知识层 - 结构化知识、事实、假设、验证结果              │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 领域知识 │ 提取事实 │ 假设断言 │ 验证结果 │ 矛盾点  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                               │
│  L1: 数据层 - 原始数据、中间计算结果、临时状态               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 原始文件 │ API响应 │ 中间计算 │ 临时变量 │ 工作进度  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                               │
└─────────────────────────────────────────────────────────────┘
                              ↑
        ┌─────────────────────┼─────────────────────┐
        │                     │                     │
   读写 │                 读写 │                 读写 │
        ▼                     ▼                     ▼
┌───────────┐          ┌───────────┐        ┌───────────┐
│  Agent A  │          │  Agent B  │        │  Agent C  │
│  业务分析  │          │  架构师    │        │ 开发工程师 │
└───────────┘          └───────────┘        └───────────┘
```

## 📋 标准数据结构

### 黑板条目标准格式（必填字段）

```json
{
  "entry_id": "BB-20260509-1430-0001",
  "layer": "L1", // L1 数据层 / L2 知识层 / L3 决策层
  "category": "api_response", // 数据分类
  "producer": {
    "agent_id": "business_analyst_001",
    "agent_role": "业务分析师",
    "produced_at": "2026-05-09T14:30:22+08:00"
  },
  "key": "user_requirements",
  "value": "结构化的用户需求数据...", // 可以是任意类型：字符串/JSON对象/数组
  "confidence": 0.95, // 置信度 0-1，人类输入=1.0，AI产出<1.0
  "version": 1,
  "status": "active", // active / superseded / deprecated / verified
  "source_ref": ["task_id_123", "conv_id_456"], // 来源追溯
  "access_control": {
    "read": ["*"], // 谁可以读，* 表示所有
    "write": ["business_analyst_001", "architect_001"] // 谁可以写
  },
  "tags": ["需求", "用户输入", "Phase1"],
  "ttl": null // 过期时间，null 表示永久有效
}
```

## 🔌 核心 API 接口（9个标准操作）

### 1. 写入操作
| API | 功能 | 说明 |
|-----|------|------|
| `write(layer, key, value, producer)` | 写入新条目 | 不覆盖，版本号自增 |
| `update(entry_id, new_value, updater)` | 更新已有条目 | 旧版本保留，标记为 superseded |
| `verify(entry_id, verifier, comment)` | 验证/确认条目 | 人类确认后 confidence = 1.0 |

### 2. 读取操作
| API | 功能 | 说明 |
|-----|------|------|
| `read(layer, key)` | 按键读取最新版本 | 返回 confidence 最高的 active 条目 |
| `read_history(layer, key)` | 读取完整历史 | 显示所有版本，包括已废弃 |
| `query_by_tag(tag)` | 按标签查询 | 跨层查询相关条目 |
| `query_by_producer(agent_id)` | 按生产者查询 | 查看某个 Agent 的所有贡献 |

### 3. 高级操作
| API | 功能 | 说明 |
|-----|------|------|
| `find_contradictions()` | 自动发现矛盾 | 检测同一键的不同值冲突 |
| `get_snapshot()` | 获取全局快照 | 用于断点续跑和状态恢复 |

## 🔄 典型协作流程（合同审查场景）

```
Step 1: 合同上传
        └─ 文件解析 Agent → 写入 L1: 原始合同文本

Step 2: 条款识别
        └─ 法务分析 Agent → 读取 L1 文本
                        → 识别出 15 个关键条款
                        → 写入 L2: 条款结构化数据

Step 3: 风险识别
        └─ 风险评估 Agent → 读取 L2 结构化条款
                        → 发现 3 个高风险条款
                        → 写入 L2: 风险评估结果

Step 4: 修改建议
        └─ 合同修改 Agent → 读取 L2 风险点
                        → 生成修改建议
                        → 写入 L2: 修改建议

Step 5: 汇总输出
        └─ 报告生成 Agent → 读取 L2 所有数据
                        → 汇总生成最终审查报告
                        → 写入 L3: 最终交付物

✅ 全程无人工干预，Agent 间完全通过黑板协作，无需点对点通信！
```

## 🧠 高级功能设计

### 1. 冲突自动检测机制
```
触发条件：同一 key 有两个不同的 value，且 confidence 都 > 0.7
自动处理流程：
  1. 标记两个条目为「待验证」状态
  2. 生成「冲突检测报告」
  3. 通知两个 Agent 进行核对
  4. 必要时升级通知 Rex 人工仲裁
  5. 仲裁结果写入黑板，confidence = 1.0
```

### 2. 断点续跑机制
```
场景：任务执行到一半，Agent 崩溃或被中断
恢复机制：
  1. 启动时调用 get_snapshot() 获取完整黑板状态
  2. 检查自己负责的条目哪些已完成
  3. 从断点处继续执行，不重复计算
  4. 新 Agent 可以无缝接手旧 Agent 的工作
```

### 3. 知识演化追踪
```
每个 key 的完整生命周期：
  原始输入 (L1) → 初步分析 (L2) → 验证确认 (L2 verified)
  → 汇总决策 (L3) → 最终交付 (L3 final)

完整可追溯：谁、在什么时间、基于什么、做了什么修改
```

## 🔐 访问控制机制

### 三层权限模型

```
L1 数据层：所有 Agent 可读写
├─ 原始数据、中间结果、工作进度
└─ 高度并发写入，版本号自动管理

L2 知识层：对应领域专家可写，所有可读
├─ 结构化知识、事实、假设、验证结果
└─ 写入需要对应的领域角色权限

L3 决策层：仅 Rex/项目经理可写
├─ 最终结论、全局决策、里程碑状态
└─ 写入必须经过 HITL 人工确认
```
