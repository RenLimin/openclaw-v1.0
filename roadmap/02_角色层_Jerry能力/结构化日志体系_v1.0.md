# 📝 结构化日志体系 v1.0

> **版本**: v1.0
> **发布日期**: 2026-05-09
> **核心目标**: 所有操作可追溯、可分析、可告警，告别黑盒日志
> **设计标准**: 完全兼容 OpenTelemetry / ELK 等主流可观测性平台

---

## 🎯 解决的核心问题

### 痛点场景
```
❌ 问题1：日志格式混乱，无法自动化解析，只能人工 grep
❌ 问题2：没有上下文关联，同一个请求的日志散落在各处，无法串联
❌ 问题3：没有分级，海量 INFO 日志淹没 ERROR 告警
❌ 问题4：没有结构化字段，无法做统计分析与可视化
❌ 问题5：日志轮转缺失，磁盘被打爆，服务崩溃
```

## 🏗️ 核心架构设计

### 7 层日志分级标准（RFC 5424 兼容）

| 级别ID | 级别名称 | 英文名 | 适用场景 | 处理策略 |
|--------|---------|-------|---------|---------|
| 0 | 紧急 | EMERG | 系统级崩溃，完全不可用 | 立即告警 + 自动熔断 |
| 1 | 告警 | ALERT | 需要立即修复的严重问题 | 5分钟内告警通知 |
| 2 | 严重 | CRITICAL | 严重功能失效，核心业务受影响 | 15分钟内告警通知 |
| 3 | 错误 | ERROR | 单个请求/操作失败，业务局部受影响 | 错误率统计 + 阈值告警 |
| 4 | 警告 | WARN | 潜在问题，不影响当前运行 | 聚合统计 + 趋势告警 |
| 5 | 通知 | NOTICE | 正常但重要的事件 | 审计留存 |
| 6 | 信息 | INFO | 正常流程记录 | 默认输出级别 |
| 7 | 调试 | DEBUG | 开发调试用详细信息 | 仅开发环境输出 |

## 📋 标准结构化字段（每条日志必含 12 个字段）

```json
{
  "timestamp": "2026-05-09T14:40:22.345+08:00",  // S1: 精确到毫秒的ISO时间戳
  "log_level": "INFO",                              // S2: 日志级别
  "log_id": "LOG-20260509-144022-A8F7X",           // S3: 全局唯一日志ID
  "trace_id": "TRACE-7A3F9C2E-541D-4B8C-9E7D",     // S4: 全链路追踪ID
  "span_id": "SPAN-12345678",                       // S5: 单步跨度ID
  "agent_id": "business_analyst_001",               // S6: 产生日志的Agent ID
  "agent_role": "业务分析师",                        // S7: Agent角色
  "task_id": "TASK-CONTRACT-REVIEW-001",           // S8: 关联任务ID
  "step_id": "S003",                                // S9: 关联步骤ID
  "event_code": "CONTRACT_PARSE_SUCCESS",           // S10: 标准事件码
  "message": "合同文件解析完成，共识别15个关键条款", // S11: 人类可读消息
  "duration_ms": 1245,                              // S12: 操作耗时（毫秒）
  "extra": {                                        // 扩展字段（任意KV）
    "file_name": "采购合同.pdf",
    "file_size_kb": 2456,
    "clause_count": 15
  }
}
```

## 🔄 日志轮转与保留策略

### 双维度轮转触发规则
| 触发维度 | 阈值 | 说明 |
|---------|------|------|
| 大小轮转 | 100MB | 单个文件达到 100MB 自动切分 |
| 时间轮转 | 每日 0 点 | 每天自动生成新的日志文件 |

### 保留策略（分级存储）
| 日志级别 | 热存储时长 | 冷存储时长 | 最终处理 |
|---------|-----------|-----------|---------|
| ERROR+ | 90 天 | 3 年 | 永久归档 |
| WARN | 30 天 | 1 年 | 清除 |
| INFO+ | 7 天 | 30 天 | 清除 |
| DEBUG | 开发环境仅保留 | 不存储 | 立即清除 |

### 文件命名规范
```
openclaw-agents-2026-05-09.1.log
         |           |       |
         |           |       └── 当日第N个分片
         |           └────────── 日期
         └────────────────────── 服务名
```

## 🔧 日志查询标准语法

### 常用查询示例
```sql
-- 1. 查询最近1小时所有ERROR日志
SELECT * FROM logs
WHERE log_level = 'ERROR'
  AND timestamp > NOW() - INTERVAL 1 HOUR
ORDER BY timestamp DESC;

-- 2. 追踪单个任务的全链路日志
SELECT * FROM logs
WHERE task_id = 'TASK-CONTRACT-REVIEW-001'
ORDER BY timestamp ASC;

-- 3. 统计各Agent的错误率排名
SELECT agent_id,
       COUNT(*) as total_logs,
       SUM(CASE WHEN log_level = 'ERROR' THEN 1 ELSE 0 END) as error_count,
       SUM(CASE WHEN log_level = 'ERROR' THEN 1 ELSE 0 END) * 100.0 / COUNT(*) as error_rate
FROM logs
WHERE timestamp > NOW() - INTERVAL 24 HOUR
GROUP BY agent_id
ORDER BY error_rate DESC;

-- 4. 慢操作TOP20
SELECT * FROM logs
WHERE duration_ms > 10000
ORDER BY duration_ms DESC
LIMIT 20;
```
