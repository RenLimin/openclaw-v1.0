# 💓 Agent健康检查与自愈机制 v1.0

> **版本**: v1.0
> **发布日期**: 2026-05-09
> **核心目标**: 7x24小时不间断运行，故障自动发现、自动恢复、自动告警
> **设计原则**: 故障检测自动化、故障恢复自动化、人工干预最小化

---

## 🎯 解决的核心问题

### 痛点场景
```
❌ 问题1：Agent进程挂了没人知道，用户反馈不能用了才发现
❌ 问题2：Agent假死（进程在但不工作），无法检测无法恢复
❌ 问题3：资源泄漏（内存/句柄），运行时间越长越慢，最终崩溃
❌ 问题4：依赖的第三方服务故障，整个系统跟着挂
❌ 问题5：单点故障，一个Agent挂了导致整个任务链卡住
```

## 📋 标准健康状态定义

### 5种健康状态码

| 状态码 | 状态名称 | 颜色 | 说明 | 处理策略 |
|--------|---------|------|------|---------|
| 0 | 健康 | 🟢 绿色 | 一切正常，负载在合理范围 | 正常运行，不干预 |

| 4 | 维护中 | 🔵 蓝色 | 人工维护/升级中 | 停止调度新任务 |

---

### L1: 保活心跳检测（1秒/次）

| 指标ID | 指标名称 | 检测方法 | 异常阈值 |
|--------|---------|---------|---------|
| L1-001 | 进程存活检测 | OS 进程表查询 | 进程不存在即异常 |
| L1-002 | PID一致性检测 | 对比注册PID与实际PID | PID不一致即异常 |
| L1-003 | 心跳上报检测 | 检查最后心跳时间 | 超过30秒无心跳 |
| L1-004 | 父进程检测 | 检查守护进程是否存活 | 守护进程不存在 |

| 指标ID | 指标名称 | 检测方法 | 异常阈值 |

| L2-001 | 任务队列长度 | 检测待处理队列长度 | 超过 100 且持续增长 |

### L3: 深度检测（1分钟/次）

| 指标ID | 指标名称 | 检测方法 | 异常阈值 |
|--------|---------|---------|---------|
| L3-001 | 探针请求响应 | 发送标准探针请求 | 3次连续失败 |
| L3-002 | 结果正确性校验 | 探针请求的输出校验 | 输出不符合预期 |
| L3-003 | 状态一致性校验 | 内存状态 vs 持久化状态 | 状态不一致 |
| L3-004 | 死锁检测 | 资源等待图分析 | 检测到循环等待 |
| L3-005 | 内部错误率 | 最近1分钟错误率 | > 20% |

### 标准探针请求设计
```python
def health_check_probe(agent_id):
    """标准健康检查探针"""
    probe_task = {
        "task_type": "HEALTH_CHECK_PROBE",
        "probe_id": generate_probe_id(),
        "timestamp": get_current_timestamp(),
        "input_data": generate_standard_test_input(),
        "expected_output_hash": calculate_hash(standard_output),
        "timeout_seconds": 30
    }

    result = send_task_to_agent(agent_id, probe_task, timeout=30)

    if not result:
        return HealthStatus.FAULT, "探针请求超时无响应"

    if result['output_hash'] != probe_task['expected_output_hash']:
        return HealthStatus.ERROR, "探针输出结果不正确"

    if result['duration_ms'] > 10000:
        return HealthStatus.SUBHEALTH, "探针响应时间超过10秒"

    return HealthStatus.HEALTHY, "探针检测正常"
```

## 🔧 五级自愈机制

### 按严重程度从低到高的恢复策略

| 级别 | 策略名称 | 触发条件 | 预期恢复时间 |
|------|---------|---------|-------------|
| R1 | 自动重试 | 单次/偶发错误 | < 1 秒 |
| R2 | 资源清理 | 内存/句柄泄漏 | < 5 秒 |
| R3 | 进程重启 | 进程异常/假死 | < 30 秒 |
| R4 | 故障转移 | 重启后仍不正常 | < 1 分钟 |
| R5 | 降级运行 | 依赖故障无法恢复 | 立即生效 |

---

### R1: 自动重试策略

```python
# 指数退避 + 抖动策略
retry_delays = [1, 2, 4, 8, 16]  # 秒
max_retries = 5

def auto_retry(func, *args, **kwargs):
    for attempt in range(max_retries):
        try:
            return func(*args, **kwargs)
        except TransientError as e:
            if attempt == max_retries - 1:
                raise e
            delay = retry_delays[attempt] + random.uniform(-0.5, 0.5)
            time.sleep(delay)
```

**适用场景**: 网络波动、API限流、临时不可用

### R2: 资源清理策略

```python
def resource_cleanup():
    """深度资源清理"""
    # 1. 清理Python对象引用
    gc.collect()

    # 2. 清理临时文件
    cleanup_temp_files()

    # 3. 关闭空闲数据库连接

```

**适用场景**: 内存泄漏、句柄泄漏、资源占用高

### R3: 进程重启策略

```python
def process_restart(agent_id):
    """优雅重启Agent进程"""
    # 1. 设置重启标志，停止接受新任务
    set_restart_flag(agent_id)

    # 2. 等待当前任务完成（最多等待30秒）
    wait_current_tasks_complete(agent_id, timeout=30)

    # 3. 持久化当前状态
    save_agent_state(agent_id)

    # 4. 关闭旧进程
    graceful_shutdown(agent_id)

    # 5. 启动新进程
    new_pid = start_new_agent_process(agent_id)

    # 6. 等待新进程就绪
    wait_agent_ready(agent_id, timeout=60)

    # 7. 恢复状态
    restore_agent_state(agent_id, new_pid)

    log.info(f"Agent {agent_id} 重启完成，新PID: {new_pid}")
```

**适用场景**: 进程假死、死锁、严重内存泄漏、无法通过R1/R2恢复

---

### R4: 故障转移策略

```
故障转移流程：
Step 1: 检测到 Agent A 连续重启3次仍不正常
Step 2: 将 Agent A 标记为故障状态，停止调度新任务
Step 3: 将 Agent A 队列中的待处理任务转移到备用 Agent B
Step 4: 启动新的 Agent C 作为新的备用
Step 5: 告警通知人工介入排查 Agent A 的根因
Step 6: Agent A 修复后重新加入集群作为备用
```

**适用场景**: 进程重启后仍无法恢复，存在持续性故障

### R5: 降级运行策略

#### 五级降级模式

| 降级级别 | 降级策略 | 影响范围 | 触发条件 |
|---------|---------|---------|---------|
| L1 | 关闭非核心功能 | 影响用户体验 | 系统负载 > 70% |
| L2 | 降低并发数 | 任务执行变慢 | 系统负载 > 80% |
| L3 | 切换备用模型 | 质量下降，成本降低 | 主模型服务故障 |
| L4 | 仅处理核心任务 | 非核心任务排队 | 系统负载 > 90% |
| L5 | 系统只读模式 | 只能查询不能修改 | 严重故障，保护数据 |

## 🚨 告警与通知机制

### 告警分级与通知方式

| 告警级别 | 触发条件 | 通知方式 | 响应SLA |
|---------|---------|---------|---------|
| P0 紧急 | 系统不可用、数据丢失风险 | 电话 + 短信 + 群@所有人 | 5分钟响应 |
| P1 严重 | 核心功能故障、性能严重下降 | 群@负责人 + 短信 | 15分钟响应 |
| P2 一般 | 非核心功能异常、性能轻微下降 | 群消息通知 | 2小时响应 |
| P3 提示 | 潜在问题、建议优化 | 日报汇总 | 24小时响应 |

### 告警收敛规则
1. **相同告警 5 分钟内只通知 1 次**，避免告警风暴
2. **连续 3 次相同告警**，自动升级一级（P2→P1→P0）
3. **故障恢复后**立即发送恢复通知，附带故障持续时间
4. **每日 8:00 汇总昨日告警总览**，生成日报
