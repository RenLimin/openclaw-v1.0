# 多智能体团队最佳实践库

## 一、架构设计最佳实践

### 1. 三层架构原则（必须遵守）
```
协调层（1人）→ 业务层（N人）→ 基础设施层（3人）
```
**为什么：**
- 协调层唯一入口，确保指令统一，避免混乱
- 业务层专业化分工，每个Agent只做自己擅长的事
- 基础设施层提供通用能力，避免重复建设

**反模式：** 扁平架构，所有Agent平等，没有协调者 → 会导致决策混乱，责任不清

### 2. 角色专业化原则（CrewAI风格）
每个Agent必须有：
- ✅ 明确的角色名称 + 标志性emoji
- ✅ 清晰的核心目标（一句话）
- ✅ 详细的背景故事（建立人设）
- ✅ 具体的技能列表（3-8项）
- ✅ 具体的工具列表（3-8项）
- ✅ 明确的决策权限边界

**为什么：** 人设越清晰，Agent的行为越可预测，协作越高效

### 3. 最小权限原则
每个Agent只能访问自己职责范围内的：
- 数据权限（不能看其他Agent的数据）
- 工具权限（不能调用其他Agent的专用工具）
- 决策权限（不能越权决策）

**为什么：** 安全第一，避免错误扩散，便于问题定位

## 二、通信协议最佳实践

### 1. 标准消息格式（JSON）
所有Agent间通信必须遵循：
```json
{
  "protocol_version": "1.0",
  "message_id": "UUID",
  "timestamp": "ISO-8601",
  "sender": "发件人",
  "receiver": "收件人 / broadcast / all",
  "message_type": "ENUM",
  "subject": "标题（100字内）",
  "content": {
    "task_description": "任务详情",
    "requirements": "具体要求",
    "deadline": "截止时间",
    "priority": "high/medium/low"
  },
  "attachments": ["附件路径"],
  "reply_to": "原消息ID",
  "trace_id": "全链路追踪ID"
}
```

### 2. 12种标准消息类型
| 类型 | 用途 | 是否需要回复 | 超时 |
|------|------|------------|------|
| TASK_ASSIGN | 分配任务 | ✅ 确认 | 1小时 |
| TASK_DELEGATE | 任务转包 | ✅ 确认 | 2小时 |
| TASK_RESULT | 结果汇报 | ✅ 确认 | 4小时 |
| DATA_REQUEST | 请求数据 | ✅ 返回 | 30分钟 |
| DATA_RESPONSE | 返回数据 | ❌ | - |
| VALIDATION_REQUEST | 请求验证 | ✅ 意见 | 2小时 |
| VALIDATION_RESULT | 返回验证 | ❌ | - |
| CONSENSUS_REQUEST | 请求共识 | ✅ 投票 | 24小时 |
| CONSENSUS_RESULT | 汇总共识 | ❌ | - |
| ERROR_REPORT | 错误上报 | ✅ 方案 | 30分钟 |
| FEEDBACK | 结果反馈 | ✅ 确认 | 24小时 |
| HEARTBEAT | 健康心跳 | ❌ | 5分钟 |

## 三、协作模式最佳实践

### 模式1：任务分配（协调层→业务层）
```
Jerry → TASK_ASSIGN → Agent
    ↓ 确认接收
Agent 执行任务
    ↓
Agent → TASK_RESULT → Jerry
    ↓ 质量校验
通过 → 闭环
不通过 → 返回修改
```

### 模式2：任务委托（业务层→基础设施层）
```
Agent A 收到任务，需要专用能力
    ↓ TASK_DELEGATE
Agent B（基础设施）执行，返回结果
    ↓
Agent A 继续完成，最终交付
```

### 模式3：交叉验证（重要任务强制）
```
Agent A 完成任务
    ↓ VALIDATION_REQUEST
Agent B 独立验证
    ↓ VALIDATION_RESULT
Agent A 根据意见修改后交付
```

**强制规则：**
- 所有对外正式报告 → 必须交叉验证
- 所有涉及资金计算 → 必须Orion验证
- 所有涉及法律内容 → 必须合规Agent验证

### 模式4：共识决策（重大问题）
```
Jerry 发现重大问题
    ↓ CONSENSUS_REQUEST → 相关所有Agent
每个Agent独立评估，返回意见
    ↓
Jerry 汇总，形成 CONSENSUS_RESULT
达成共识 → 执行
未达成 → 再次讨论或升级
```

### 模式5：端到端流水线（复杂任务）
```
任务 → Agent1 → Agent2 → Agent3 → ... → 最终交付
    ↓         ↓         ↓
  质量校验  质量校验  质量校验
```

## 四、成本优化最佳实践

### 1. Orion成本调度策略
- 简单任务 → 用小模型（GPT-3.5/LiteLLM）
- 复杂任务 → 用大模型（GPT-4/Claude）
- 代码生成 → 用专门的代码模型
- 批量任务 → 用批处理+缓存

### 2. 脚本优先原则
> 凡是脚本/程序能搞定的，绝对不用大模型

**优先级：**
1. 本地脚本（Python/Bash）→ 成本≈0
2. 本地数据库查询 → 成本≈0
3. API调用（非LLM）→ 成本低
4. 小模型调用 → 成本中
5. 大模型调用 → 成本高

**目标：** 大模型调用占比 < 20%

## 五、安全闸门最佳实践

### 1. 三层安全闸门
```
高风险操作
    ↓
第1层：Agent自检（权限检查）
    ↓
第2层：Jerry审批（协调层审核）
    ↓
第3层：用户确认（最终人在回路）
    ↓
执行
```

### 2. 高风险操作清单
- 修改生产环境数据
- 发送对外正式邮件/消息
- 执行大额资金操作
- 修改系统核心配置
- 删除重要数据
- 对外发布内容

以上操作必须经过完整三层审批！
