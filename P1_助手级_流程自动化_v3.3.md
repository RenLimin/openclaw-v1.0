# 🤝 P1 助手级：流程自动化

> **通用目标：自动化完成 80% 重复性工作，7×24 小时值守**
> **核心价值：降本、增效、高可用**
> **大模型调用占比：< 15%**
> **建议周期：2-3 周**

---

## 📋 适用场景

- 客服机器人
- 自动化运维
- 定时任务处理
- 消息通知推送
- 小团队协作助手

---

## 🏗️ MCP 四层控制架构（通用）

**Model Control Plane - 拦住 80% 不必要的大模型调用**

```
用户请求
    ↓
┌─────────────────────────────────────────────────┐
│  1. 路由层：简单任务脚本搞定                       │
│     目标：脚本解决率 ≥ 70%                        │
│     逻辑：关键词匹配 → 规则引擎 → 脚本执行        │
├─────────────────────────────────────────────────┤
│  2. 缓存层：相同问题直接返回                       │
│     目标：缓存命中率 ≥ 50%                        │
│     实现：SQLite + LRU 淘汰策略                  │
├─────────────────────────────────────────────────┤
│  3. 熔断层：错误/超限时自动降级                    │
│     目标：可用性 ≥ 99.5%                         │
│     策略：连续 5 次错误 → 降级到纯脚本模式        │
├─────────────────────────────────────────────────┤
│  4. 预算层：控制调用量，防止成本失控               │
│     目标：月度成本偏差 < 10%                      │
│     告警：用到 50% 预警 → 80% 告警 → 95% 限流   │
└─────────────────────────────────────────────────┘
```

### 通用成本公式

```
大模型实际成本 = 理论成本 × 脚本解决率 × (1 - 缓存命中率)
```

**P1 优化目标：让成本降为原来的 15%！**

## 👥 P1 人机协作模式：助手模式

```
用户
  ↓ 提需求
智能体执行
  ↓ 交结果
用户确认/修改
```

**核心原则：智能体做执行，人做确认！**

### 助手模式典型场景

| 场景 | 人做什么 | 智能体做什么 |
|------|---------|-------------|
| 写报告 | 确认提纲、审核修改 | 撰写初稿、格式调整 |
| 数据分析 | 确认分析维度、解读结论 | 数据提取、计算、可视化 |
| 客服回复 | 复杂问题人工介入 | 标准问题自动回复、知识库检索 |
| 运维操作 | 关键操作二次确认 | 脚本执行、日志分析、告警通知 |

## 🔐 新增：第9道风控 - 自动作风控闸门体系

### 5种通用安全闸门模式

| 闸门模式 | 触发条件 | 执行流程 |
|---------|---------|---------|
| **G1 预览确认** | 所有写入操作默认 | ① 生成操作预览 → ② 展示影响范围 → ③ 用户确认后执行 |
| **G2 沙箱预演** | 修改/删除/批量操作 | ① 测试环境先执行 → ② 展示结果对比 → ③ 用户确认后生产执行 |
| **G3 白名单限制** | 系统级/高风险操作 | ① 检查操作是否在白名单 → ② 不在白名单直接拒绝 |
| **G4 渐进式执行** | 高风险批量操作 | ① 先执行1个 → ② 展示结果 → ③ 用户确认 → ④ 再执行剩余 |
| **G5 零信任原则** | 敏感凭证/密钥操作 | ① 默认禁止输出 → ② 用户明确说明用途 → ③ 最小范围展示 |

### 通用风险矩阵对照表

| 资源类型 \ 操作类型 | 只读O1 | 写入O2 | 修改O3 | 删除O4 | 批量O5 | 全覆盖O6 |
|-------------------|--------|--------|--------|--------|--------|---------|
| **R1 代码仓库** | 🟢 G1 | 🟡 G1 | 🟠 G2 | 🔴 G2 | 🔴 G2 | 🔴 G3 |
| **R2 本地文件** | 🟢 G1 | 🟡 G1 | 🟠 G1 | 🔴 G1 | 🔴 G2 | 🔴 G3 |
| **R3 业务系统** | 🟢 G1 | 🟠 G1 | 🟠 G2 | 🔴 G2 | 🔴 G4 | 🔴 G3 |
| **R4 消息系统** | 🟢 G1 | 🟠 G1 | 🟠 G1 | 🔴 G1 | 🔴 G2 | ❌ 禁止 |
| **R5 计算资源** | 🟢 G1 | 🟠 G1 | 🟠 G2 | 🔴 G2 | 🔴 G4 | ❌ 禁止 |
| **R6 敏感凭证** | 🟡 G5 | 🔴 G3 | 🔴 G3 | ❌ 禁止 | ❌ 禁止 | ❌ 禁止 |
| **R7 外部API** | 🟢 G1 | 🟡 G1 | 🟠 G1 | 🔴 G1 | 🔴 G2 | 🔴 G3 |

### Git操作专项闸门（从本次错误中学习）

```
🔴 触发条件：任何 git push / git push --force / git reset --hard

✅ 强制执行步骤：
1. 用户说"新仓库/重建/重新提交" → 自动执行 rm -rf .git && git init
2. git status → 列出所有文件数量和名称
3. git diff --cached --stat → 列出变更行数
4. 给用户展示以上信息 + 明确问："确认提交并push吗？"
5. 只有用户明确回复"✅ 确认"或"执行"才可以push
```

## 🕳️ P1 阶段四大陷阱

| 陷阱 | 表现 | 避坑指南 |
|------|------|---------|
| **MCP 缺失症** | 不做成本控制，大模型随便调用 | **预算先行**：P1 第一天就必须上 MCP 四层 |
| **重开发轻运营** | 上线就不管了，没人跟进 | **运营前置**：P1 开始就指定运营负责人，每周周报 |
| **数据裸奔** | 敏感数据直接投喂，不做隔离 | **安全底线**：P1 必须上沙箱和数据脱敏 |
| **流程僵化** | 什么都想自动化，包括低频流程 | **二八原则**：只自动化 20% 高频流程，带来 80% 收益 |
