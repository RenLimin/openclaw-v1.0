## 🔴 0 级强制规则：工作目录限制

> 🚨 **最高优先级，所有 Agent 必须严格遵守**

### 0.1 工作目录限制规则

**所有 OpenClaw 操作仅限在 `~/.openclaw` 目录（即 `/Users/bangcle/.openclaw`）内进行，除非 Rex 单独发送明确指令授权。**

| 操作类型 | 允许目录 | 例外条件 |
|---------|---------|---------|
| **文件读取** | `~/.openclaw/**` | Rex 明确指定读取其他目录 |
| **文件写入** | `~/.openclaw/**` | Rex 明确指定写入其他目录 |
| **文件删除** | `~/.openclaw/**` | Rex 明确指定删除其他目录 + 二次确认 |
| **脚本执行** | `~/.openclaw/**` | Rex 明确指定执行其他目录脚本 |
| **Git 操作** | `~/.openclaw/workspace/` 及其子目录 | Rex 明确指定其他 Git 仓库 |
| **配置修改** | `~/.openclaw/**` | 无例外 |

### 0.2 违规后果

- 任何 Agent 违反此规则，Jerry 有权立即终止其执行
- 违规操作造成的数据损失由违规 Agent 负责回滚恢复
- 多次违规将触发 Agent 重新训练流程

### 0.3 合规检查流程

1. 执行任何文件/目录操作前，先检查路径是否在 `~/.openclaw` 内
2. 如果路径不在允许范围内，立即停止并询问 Rex 是否授权
3. 获得明确授权后，记录授权信息再执行操作
4. 所有跨目录操作必须在日志中明确记录授权来源

### 0.4 业务目标分析前置规则

**创建任何智能体团队之前，必须先完成业务目标深度分析并获得用户确认。**

**标准三阶段分析流程：**

```

   ├── 核心目标识别（5个问题必须回答）
   │   ├── 要解决的核心问题是什么？
   │   ├── 谁是最终受益者/用户？
   │   ├── 成功的具体衡量标准是什么？
   │   ├── 有什么时间限制/里程碑要求？
   │   └── 有什么预算/资源约束？
   ├── 利益相关者映射（影响力/兴趣矩阵）
   └── 风险与约束识别（10项通用风险检查）
        ↓
阶段二：能力映射（必须完成）
   ├── 核心能力需求矩阵（三级优先级：必须/应该/可以）
   ├── 角色推导与职责划分（4原则）
   └── 协作流程设计
        ↓
阶段三：方案输出（必须完成）
   ├── 团队架构蓝图（三层架构）
   ├── 成功指标体系（三层KPI）
   ├── 四阶段实施路线图
   └── ROI预估与投资回报分析
```

**标准交付物（7份，缺一不可）：**
1. 业务目标分析报告
2. 团队架构蓝图
3. 角色定义集
4. 成功指标体系
5. 实施路线图
6. ROI分析报告
7. 风险评估与应对

**用户确认流程（强制执行，严禁跳过）：**
```
完成全部分析并生成7份交付物
    ↓
向用户展示完整方案
    ↓
根据意见迭代修改（可多轮）
    ↓
用户明确表示「确认」「同意」「可以开始」
    ↓
✅ 记录确认信息（时间、用户、确认内容）
    ↓
```

**使用工具：** `python skills/agent-team-creator/scripts/goal_analyzer.py`

**质量验收标准：**
- ≥ 90分：优秀，可直接进入创建流程
- ≥ 70分：良好，需要补充细节后确认
- < 70分：不合格，需要重新分析

**违规后果：**
- 未完成分析直接创建团队，立即终止所有进程
- 未获得用户确认直接执行，立即回滚所有操作
- 多次违反将触发 Jerry 重新训练流程

### 1.1 Agent 职责划分

| 原则 | 说明 | 示例 |

| 原则 | 说明 | 示例 |
|------|------|------|
| **全局共享** | 通用技能放 `workspace/skills/` | tavily-search, summarize |

### 1.3 重复处理策略

```
发现重复功能
   │
   ▼
1. 评估哪个实现更优
   │
   ├── 现有更优 → 删除新的，复用现有
   │
   ▼ 新的更优
2. 替换旧的，更新依赖
   │
   ▼ 各有优势
3. 整合为一个，保留最佳特性
```

#### 2.3.3 风控规避规则

| 规则 | 具体要求 |
|------|----------|
| **异常模式禁止** | 禁止短时间内大量相似请求、循环请求、死循环调用、批量自动化请求 |
| **内容安全** | 所有输入输出必须符合中国法律法规，禁止生成敏感、违法、违规、色情、暴力、政治敏感内容 |
| **"人类行为"模拟** | 请求应模拟人类使用模式：有思考间隔、有停顿、不连续高频 |
| **失败重试** | 失败后指数退避重试：2s → 4s → 8s → 16s，最多 3 次 |
| **429 处理** | 遇到 429 限流，强制等待 300 秒（5 分钟）后再重试，且降低并发 50% |
| **401/403 处理** | 遇到认证失败，**立即停止所有 Provider**，上报 Rex，不得继续任何调用 |
| **错误率监控** | 10 分钟内错误率 > 20% → 自动熔断该 Provider 1 小时 |
| **敏感词检测** | 所有 outgoing 请求前，所有 incoming 响应后，都必须进行敏感词扫描 |

#### 2.3.7 Provider 智能路由策略 v2.0

**首选 → 备选 → 降级 → 更降级 → 最后手段**

**切换规则**：
- Volcengine 当日调用 > 200 次 → 自动切换到 DeepSeek
- 当前 Provider 不可用时自动切换到下一级

- bailian 连续 3 次失败 → 立即熔断 24 小时，不得重试

##### 第三步：调用行为注入（人类模拟）

| 行为优化项 | 规则 |
|-----------|------|
| **调用前延迟** | 所有 Provider 调用前注入 1-3s 随机延迟（DeepSeek 可 0.5-2s） |
| **连续调用停顿** | 每 3 次连续调用后，强制停顿 5-10s |
| **错误重试** | 失败后 2-5s 随机延迟重试，不是固定指数退避 |
| **时间窗口** | Volcengine/bailian 仅在 9:00-18:00 使用，其他时间自动切 DeepSeek |

## 三、错误处理

### 3.1 重试策略

- 每个 Agent 遇到失败先自动重试（最多 3 次）
- 3 次失败后上报 Jerry
