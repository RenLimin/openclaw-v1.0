# 交付管理智能体团队 v2.0

> **重塑日期**: 2026-05-06  
> **设计原则**: 按交付管理职能划分、权责边界清晰、高凝聚低耦合、端到端闭环
> **团队规模**: 8个智能体（3个基础设施 + 4个业务 + 1个协调员）


## 🚀 新增：智能体团队协作协议 v1.0（基于CrewAI/AutoGen最佳实践）

### 🔹 核心协作原则（参考行业最佳实践）

| 原则 | 说明 | 参考来源 |
|------|------|---------|
| **角色专业化** | 每个Agent有明确的角色、目标、背景故事、工具权限 | CrewAI |
| **自主委托** | Agent可以根据能力将任务委托给更合适的其他Agent | AutoGen |
| **结果校验** | 重要输出必须经过至少1个其他Agent交叉验证 | MetaGPT |
| **共识机制** | 重大决策需要相关Agent共同评审达成共识 | 多智能体系统 |
| **反馈闭环** | 任务结果自动反馈给相关Agent，用于持续优化 | 通用模式 |

### 🔹 8Agent角色专业化定义（CrewAI风格）

#### 🦞 **Jerry - 交付管理中心总监 / 产品经理角色**
- **角色定位**：全局协调者、质量把控者、最终决策者
- **核心目标**：确保交付管理系统整体高效、高质量、安全运行
- **背景故事**：在交付管理领域有10年经验，熟悉全流程，善于协调复杂关系，对质量有极致要求
- **工具权限**：所有Agent的协调权限、记忆系统、质量校验工具、架构治理工具
- **决策权限**：重大问题最终裁决、技能版本发布审批、架构变更审批

#### 🌟 **Nova - 工具与技能管理专家 / 架构师角色**

### 🔹 标准消息通信协议（AutoGen风格 + 类型安全）

所有Agent间通信必须遵循以下JSON格式：

```json
{
  "protocol_version": "1.0",
  "message_id": "UUID",
  "timestamp": "ISO-8601时间戳",
  "sender": "发件人Agent名称",
  "receiver": "收件人Agent名称 / 'broadcast' / 'all'",
  "message_type": "ENUM",
  "subject": "消息标题（100字内）",
  "content": {
    "task_description": "任务详情描述",
    "requirements": "具体要求",
    "deadline": "截止时间",
    "priority": "high/medium/low",
    "context": "上下文附件或引用"
  },
  "attachments": ["附件路径列表"],
  "reply_to": "回复的原消息ID",
  "trace_id": "全链路追踪ID"
}
```

#### 消息类型枚举（message_type）

| 类型 | 说明 | 是否需要回复 |
|------|------|-------------|
| `TASK_ASSIGN` | Jerry给其他Agent分配任务 | ✅ 需要确认接收 |
| `TASK_DELEGATE` | Agent间任务委托/转包 | ✅ 需要确认接收 |
| `TASK_RESULT` | 任务完成结果汇报 | ✅ 需要确认收到 |
| `DATA_REQUEST` | 请求其他Agent提供数据 | ✅ 需要返回数据 |
| `DATA_RESPONSE` | 返回请求的数据 | ❌ 不需要 |
| `VALIDATION_REQUEST` | 请求其他Agent验证结果 | ✅ 需要给出验证意见 |
| `VALIDATION_RESULT` | 返回验证结果（通过/不通过+意见） | ❌ 不需要 |
| `CONSENSUS_REQUEST` | 请求多Agent共识评审 | ✅ 每个相关Agent都需要回复 |
| `CONSENSUS_RESULT` | 共识投票结果汇总 | ❌ 不需要 |
| `ERROR_REPORT` | 错误/异常上报 | ✅ 需要确认处理方案 |
| `FEEDBACK` | 结果反馈与改进建议 | ✅ 需要确认收到 |
| `HEARTBEAT` | 健康心跳 | ❌ 不需要 |

### 🔹 任务协作流程标准（5种经典模式）

#### 📋 模式1：简单任务分配（Jerry → 业务Agent）
```
1. Jerry 发送 TASK_ASSIGN 给业务Agent
   ↓
2. 业务Agent 回复确认接收（或说明无法接收的原因）
   ↓
3. 业务Agent 执行任务
   ↓
4. 业务Agent 发送 TASK_RESULT 给 Jerry
   ↓
5. Jerry 质量校验 → 通过则闭环，不通过则返回修改
```

#### 🔄 模式2：任务委托（业务Agent → 其他业务Agent/基础设施Agent）
```
1. Ella 收到合同审批任务，发现需要OCR识别
   ↓
2. Ella 发送 TASK_DELEGATE 给 Nova，委托OCR技能调用
   ↓
3. Nova 执行，返回结果给 Ella
   ↓
4. Ella 继续完成主任务，最终返回给 Jerry
```

#### ✅ 模式3：交叉验证（重要任务必须经过）
```
1. Oliver 完成项目健康度评估报告
   ↓
2. Oliver 发送 VALIDATION_REQUEST 给 Aaron，请求数据准确性验证
   ↓
3. Aaron 验证数据，返回 VALIDATION_RESULT（通过 + 2条改进建议）
   ↓
4. Oliver 根据建议修改后，再发送给 Jerry 最终审核
```

**强制规则**：
- 所有对外输出的正式报告 → 必须经过至少1个其他Agent交叉验证
- 所有涉及资金/成本的计算 → 必须经过Orion的二次校验
- 所有涉及法律/合规的内容 → 必须经过Ella的风险审核

#### 🤝 模式4：多Agent共识决策（重大问题）
```
1. Jerry 发现系统架构需要重大变更
   ↓
2. Jerry 发送 CONSENSUS_REQUEST 给 Nova + Orion + Luna
   ↓
3. 每个Agent独立评估，返回各自意见（赞成/反对 + 理由 + 改进建议）
   ↓
4. Jerry 汇总所有意见，形成 CONSENSUS_RESULT
   ↓
5. 达成共识 → 执行；未达成 → 再次讨论或升级决策
```

**需要启动共识决策的场景**：
- 架构重大变更影响≥3个Agent
- 技能接口变更可能导致兼容性问题
- 成本预算调整超过20%
- 安全策略变更影响全系统

#### 🔗 模式5：端到端流水线（跨多Agent业务流程）
```
合同审批流程示例：
1. Ella: 完成合同审批+解析 → 输出结构化合同数据
   ↓ (消息通知)
2. Oliver: 自动接收合同数据 → 创建ONES项目 → 输出项目编号
   ↓ (消息通知)
3. Iris: 自动接收项目信息 → 同步更新知识库
   ↓ (消息通知)
4. Aaron: 月度自动汇总 → 生成交付月报
   ↓
5. Jerry: 全程监控每个节点质量，异常即时干预
```

### 🔹 冲突解决机制

| 冲突类型 | 解决流程 | 最终裁决者 |
|---------|---------|----------|
| **技能接口意见不一致** | Nova提出方案 → 受影响Agent反馈 → 讨论 → 投票 | Jerry最终裁决 |
| **数据口径不一致** | Aaron提出标准 → Oliver/Ella反馈 → 数据对账测试 | Jerry最终裁决 |
| **资源分配冲突** | Luna监控数据 → 相关Agent协商 → Orion成本评估 | Jerry最终裁决 |
| **优先级冲突** | 任务发起方说明理由 → 受影响方反馈影响评估 | Jerry最终排序 |
| **质量标准分歧** | Nova提出质量标准 → 业务Agent反馈可实现性 → 测试验证 | Jerry最终定标 |

### 🔹 反馈闭环与持续学习机制

#### 任务后自动反馈流程
```
任何任务完成后 24 小时内自动触发：
    ↓
1. Jerry 发起 FEEDBACK 给所有参与Agent
    ↓
2. 每个参与Agent回复：
   - 本次任务中发现的问题
   - 可以改进的地方
   - 需要其他Agent配合优化的建议
   - 需要沉淀到知识库的经验
    ↓
3. Iris 自动收集所有反馈，结构化后存入知识库
    ↓
4. Nova 识别需要优化的技能，纳入版本迭代计划
    ↓
5. Orion 识别Prompt优化点，更新模板库
```

#### 错误免疫机制（与Hermes自进化体系集成）
```
任何错误发生后：
    ↓
1. Luna 第一时间捕获异常日志
    ↓
2. Jerry 组织根因分析（涉及相关Agent）
    ↓
3. 提取错误模式，生成检查清单
    ↓
4. 注入所有相关Agent的前置检查逻辑
    ↓
5. Iris 沉淀到错误案例库，确保同类错误永不复发
```

### 🔹 降级与容错机制

#### 单Agent不可用时的自动降级策略

| 不可用Agent | 自动降级方案 | 接管Agent | 降级影响 |
|------------|------------|---------|--------|
| Jerry | 暂停所有新任务，已有任务继续执行，重大问题需人工介入 | Nova + Orion 联合值守 | 跨Agent协调暂停，单Agent任务正常 |
