## 📋 项目概述

### 项目背景
当前CISSP知识库已完成基础架构搭建（143个文档元数据入库），但内容提取、语义检索、Agent集成等核心能力尚未开发。需要严格按照智能体进阶规划的最新标准（第20/21/22章）完成开发。

### 建设目标
1. ✅ **内容深化**：从"元数据仓库"升级为"真正可用的知识库"
2. ✅ **向量增强**：从"关键词检索"升级为"语义理解检索"
3. ✅ **智能体集成**：从"独立知识库"升级为"Agent可调用的知识服务"

### 预期收益
- 检索命中率从 <10% → >85%
- RAG回答可追溯、可验证
- 所有Agent可无缝调用知识库

## 🔴 **Phase 0: 内容深化（P0，核心基础）
**目标：从「元数据仓库」→「内容可用的知识库」**
**计划周期：4天（正常）/ 3天（乐观）/ 6天（保守）**

| 任务ID | 任务名称 | 预估工时 | 优先级 | 依赖 | 验收标准 |
|--------|---------|---------|--------|------|---------|
| P0-T01 | PDF批量内容深度提取 | 1.5天 | 🔴 P0 | PyPDF2已安装 | 101个PDF文本提取率 >90% |
| P0-T02 | DOCX/MD/TXT文本解析 | 0.5天 | 🔴 P0 | python-docx | 22个DOCX + 其他文本格式100%解析 |
| P0-T03 | 语义智能分块算法实现 | 1.5天 | 🔴 P0 | - | 按标题层级切分，重叠率20%，分块大小500-1500tokens |
| P0-T04 | 全文索引重建 & 质量校验 | 0.5天 | 🔴 P0 | - | 去重、完整性检查、分块质量抽检合格率 >90% |

### Phase 0 关键里程碑
- [ ] 143个文档内容100%提取完成
- [ ] 智能分块算法上线，生成 ~5000-10000个高质量分块
- [ ] FTS5全文索引重建完成
- [ ] 关键词检索验证："CISSP安全域"等10个测试查询命中率 >90%

### Phase 0 风险点与应对
- ⚠️ 风险：PDF中文编码问题（UniGB-UTF16-H等编码不支持）
  - 应对：降级策略，先用PyPDF2提取可用部分，标记问题PDF后续人工处理
- ⚠️ 风险：扫描版PDF无法提取文本
  - 应对：标记为"需要OCR处理"，后续批次解决

## 🟡 **Phase 1: 向量增强（P1，质量提升）**
**目标：从「关键词检索」→「语义理解检索」**
**计划周期：5.5天（正常）/ 3.5天（乐观）/ 8天（保守）**

| 任务ID | 任务名称 | 预估工时 | 优先级 | 依赖 | 验收标准 |
|--------|---------|---------|--------|------|---------|
| P1-T01 | 本地Embedding模型接入 | 1天 | 🔴 P0 | 硬盘空间≥10GB | bge-m3模型加载成功，单条向量化耗时<100ms |
| P1-T02 | 批量向量化处理 | 1.5天 | 🟡 P1 | P1-T01完成 | 所有分块完成向量化，进度条可监控 |
| P1-T03 | FAISS向量索引构建 | 0.5天 | 🟡 P1 | FAISS库安装 | 索引构建完成，检索速度<50ms/query |
| P1-T04 | 混合检索算法实现 | 1.5天 | 🟡 P1 | P0-T04 + P1-T03 | 全文+向量融合检索，权重可配置 |
| P1-T05 | 检索质量指标体系实现 | 1天 | 🟡 P1 | - | Recall/MRR/NDCG自动化计算 |

### Phase 1 关键里程碑
- [ ] bge-m3中文Embedding模型部署完成
- [ ] 所有分块完成向量化（~5000-10000向量）
- [ ] FAISS IndexIVFFlat索引构建完成
- [ ] 混合检索API可用，支持关键词/语义/混合三种模式
- [ ] 质量指标自动化计算：Recall@10 >85%, MRR >0.7

### Phase 1 风险点与应对
- ⚠️ 风险：bge-m3模型太大，内存不足
  - 应对：降级使用bge-small-zh模型（~100MB），后续升级
- ⚠️ 风险：向量化速度太慢（CPU模式）
  - 应对：多进程并行处理，分批提交

## 🧪 测试策略

### 单元测试（每个模块必做）
- [ ] PDF提取：对10个样本PDF做准确率测试
- [ ] 分块算法：边界检查、重叠检查、大小检查
- [ ] 向量化：相似度验证（相同内容相似度>0.95）
- [ ] 检索API：接口响应格式、错误处理、边界条件

### 集成测试（阶段验收必做）
- [ ] 端到端：文档上传 → 提取 → 分块 → 向量化 → 检索 → RAG回答
- [ ] 性能测试：100并发查询响应时间
- [ ] 长稳测试：连续运行24小时无崩溃

### 质量验收标准（必须达标）
| 指标 | 目标值 | 测量方法 |
|------|--------|---------|
| 内容提取率 | >90% | 抽检20个文档 |
| Recall@10 | >85% | 50个标准查询集 |
| MRR | >0.7 | 同上 |
| 查询响应 P95 | <300ms | 100次查询统计 |
| RAG答案可追溯率 | 100% | 每条回答必须标注来源 |

## 🚀 并行加速策略（可缩短30%+时间）

### 策略1：Phase 0 & Phase 1部分并行
- Phase 0 处理第一批PDF时，同步部署Embedding模型
- 预计节省：1-2天

### 策略2：多进程批量处理
- PDF提取：4进程并行
- 向量化：8进程并行（CPU密集型）
- 预计节省：1-2天

### 策略3：边做内容边做API
- 有第一批分块数据就开始开发API
- 不用等所有数据处理完
- 预计节省：1-2天

**采用全部3种策略：总周期可压缩至 8-10天**
