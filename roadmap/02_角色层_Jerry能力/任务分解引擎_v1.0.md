# 🎯 任务分解引擎 v1.0

> **版本**: v1.0
> **发布日期**: 2026-05-09
> **核心能力**: 复杂业务目标 → 结构化可执行任务清单
> **设计目标**: 任务分解准确率 ≥ 80%

---

## 🧠 任务分解五步法算法

```
用户输入原始目标
    ↓
Step 1: 目标理解与边界定义
    ↓
Step 2: 任务分层拆解（WBS 工作分解结构）
    ↓
Step 3: 依赖关系识别与排序
    ↓
Step 4: 验收标准定义
    ↓
Step 5: 风险识别与缓解建议
```

---

## 📋 Step 1: 目标理解与边界定义

### 标准分析框架（5W2H）

| 维度 | 问题 | 目的 |
|------|------|------|
| **WHAT** | 具体要做什么？核心交付物是什么？ | 明确定义范围 |
| **WHY** | 为什么要做？要解决什么问题？ | 理解业务价值 |
| **WHO** | 干系人有谁？谁使用？谁验收？ | 识别角色 |
| **WHEN** | 时间要求？有什么 deadline？ | 时间约束 |
| **WHERE** | 在什么环境运行？部署在哪？ | 环境约束 |
| **HOW** | 有什么技术/方法要求？ | 技术约束 |
| **HOW MUCH** | 成本/资源限制？ | 资源约束 |

### 输出：目标摘要卡
```json
{
  "goal_summary": "一句话描述核心目标",
  "deliverables": ["交付物1", "交付物2"],
  "constraints": {
    "time": "时间限制",
    "tech": "技术限制",
    "resource": "资源限制"
  },
  "success_criteria": "成功的标准是什么"
}
```

## 🏗️ Step 2: 任务分层拆解（WBS 工作分解）

### 三层拆解标准

| 层级 | 名称 | 时间范围 | 特点 |
|------|------|---------|------|
| **L1 阶段** | Phase | 1-2 周 | 可独立交付的大里程碑 |
| **L2 任务** | Task | 1-3 天 | 完整的可验收单元 |
| **L3 子任务** | Subtask | 1-4 小时 | 单人可独立完成的原子单元 |

### 拆解原则（强制执行）

1. **MECE 原则**: 相互独立，完全穷尽，不重不漏
2. **80小时法则**: 任何 L2 任务不超过 80 小时工作量
3. **8小时法则**: 任何 L3 子任务不超过 8 小时工作量
4. **单一责任人**: 每个 L2/L3 任务有且只有一个责任人

### 输出：任务树结构
```json
{
  "phase_id": "P1",
  "phase_name": "核心功能补全",
  "tasks": [
    {
      "task_id": "T1.1",
      "task_name": "人类反馈机制设计",
      "estimated_hours": 4,
      "subtasks": [
        {"id": "T1.1.1", "name": "5类场景定义", "hours": 1},
        {"id": "T1.1.2", "name": "三阶段流程设计", "hours": 1.5},
        {"id": "T1.1.3", "name": "回滚机制设计", "hours": 0.5},
        {"id": "T1.1.4", "name": "文档编写", "hours": 1}
      ]
    }
  ]
}
```

## 🔗 Step 3: 依赖关系识别与拓扑排序

### 4 种标准依赖类型

| 依赖类型 | 代码 | 说明 | 图示 |
|---------|------|------|------|
| **完成-开始** | FS | A 完成后 B 才能开始 | A → B |
| **开始-开始** | SS | A 开始后 B 才能开始 | A↗B |
| **完成-完成** | FF | A 完成后 B 才能完成 | A↘B |
| **开始-完成** | SF | A 开始后 B 才能完成 | A↗↘B |

### 依赖发现启发式规则

```
规则 1: 设计 → 开发 → 测试 → 部署 （经典瀑布流，FS）
规则 2: 基础模块 → 上层应用 （自底向上，FS）
规则 3: 数据准备 → 分析 → 报告 （数据流，FS）
规则 4: 并行任务都完成 → 汇总 （汇合点，FS）
规则 5: API 定义 → 前后端并行开发 （SS，接口定义后可并行）
```

### 输出：有向无环图 (DAG) + 关键路径
```json
{
  "dependencies": [
    {"from": "T1.1", "to": "T1.2", "type": "FS"},
    {"from": "T1.2", "to": "T1.3", "type": "FS"}
  ],
  "critical_path": ["T1.1", "T1.2", "T1.3"],
  "total_duration_hours": 120,
  "topological_order": ["T1.1", "T1.2", "T1.3"]
}
```

## ✅ Step 4: 验收标准定义

### 验收三级标准

| 级别 | 名称 | 说明 | 量化指标 |
|------|------|------|---------|
| **P0** | 必须有 | 缺了就不算完成 | 100% 实现 |
| **P1** | 应该有 | 重要但不阻塞 | ≥ 90% 实现 |
| **P2** | 可以有 | 锦上添花，后续优化 | ≥ 70% 实现 |

### 验收标准模板
```json
{
  "acceptance_criteria": [
    {
      "level": "P0",
      "criteria": "5类必须人工介入的场景完整定义",
      "validation_method": "文档审查",
      "target_coverage": "100%"
    }
  ]
}
```

## ⚠️ Step 5: 风险识别与缓解建议

### 风险矩阵

| 风险等级 | 发生概率 | 影响程度 | 应对策略 |
|---------|---------|---------|---------|
| 🔴 高 | ≥ 70% | 严重 | 立即缓解，重点监控 |
| 🟡 中 | 30-70% | 中等 | 制定预案，定期检查 |
| 🟢 低 | < 30% | 轻微 | 接受，偶尔检查 |

### TOP 10 常见风险清单（自动检查）
```
1. 需求不明确，范围蔓延风险 🔴
2. 技术难度超预期，存在未知领域 🟡
3. 依赖的外部资源/接口不可靠 🟡
4. 时间估算过于乐观 🟡
5. 人员能力不匹配 🟢
6. 沟通协作成本高 🟢
7. 测试环境不就绪 🟡
8. 性能/安全等非功能需求遗漏 🟡
9. 数据质量/可用性问题 🟡
10. 架构设计缺陷导致后期重构 🔴
```
