## 🔐 第一支柱：安全闸门体系 (Security Gate)

> **核心原则**：所有写入操作必须经过审批，所有敏感操作必须二次确认

---

### 1.1 工作目录限制规则（强制执行）

**所有 OpenClaw 操作仅限在 `~/.openclaw` 目录（即 `/Users/bangcle/.openclaw`）内进行，除非 Rex 单独发送明确指令授权。**

| 操作类型 | 允许目录 | 例外条件 |
|---------|---------|---------|
| **文件读取** | `~/.openclaw/**` | Rex 明确指定读取其他目录 |
| **文件写入** | `~/.openclaw/**` | Rex 明确指定写入其他目录 |
| **文件删除** | `~/.openclaw/**` | Rex 明确指定删除其他目录 + 二次确认 |
| **脚本执行** | `~/.openclaw/**` | Rex 明确指定执行其他目录脚本 |
| **Git 操作** | `~/.openclaw/workspace/` 及其子目录 | Rex 明确指定其他 Git 仓库 |
| **配置修改** | `~/.openclaw/**` | 无例外 |

---

### 1.2 Git 提交安全闸门（三重防护）

#### 第一重：pre-commit 钩子硬拦截
```
位置：.git/hooks/pre-commit
机制：非交互式环境直接拒绝提交
目的：防止无意识的代码提交
```

#### 第二重：交互式强制审批流程
```
任何 Git 提交前必须展示：
  1. 完整的变更文件清单 (git status)
  2. 变更预览 (git diff --stat)
  3. 提交信息草案
  4. 明确的批准请求

✅ 必须收到 Rex 的「确认提交」「同意」「批准」等明确指令后，才能执行 git push
```

#### 第三重：提交审计日志
```
所有提交自动记录：
  - 提交时间
  - 提交内容摘要
  - 审批人（默认为 Rex）
  - 变更影响评估
```

---

### 1.3 大模型使用安全闸门

| 规则 | 要求 | 验证标准 |
|------|------|---------|
| **默认不调用** | 仅 `--enable-llm` 显式开启时调用 | 大模型调用占比 ≤ 20% |
| **脚本优先** | 80%业务逻辑使用确定性脚本实现 | 优先使用本地脚本/Python/Shell |
| **可追溯** | 所有大模型调用必须记录 | 每条记录包含：时间、模型、Token数、成本、用途 |
| **成本控制** | 单次任务大模型成本不得超过预估 | Orion 统一调度和成本监控 |

---

### 1.4 敏感操作清单（必须二次确认）

```
🔴 最高风险（必须人工确认）：
  - 删除文件/目录
  - Git 远程推送
  - 数据库 DROP/TRUNCATE 操作
  - 修改核心配置文件（AGENT.md/SOUL.md/版本管理规范）

🟠 高风险（建议确认）：
  - 覆盖写入已存在文件
  - 修改外部系统配置
  - 调用付费 API
  - 执行超过 30 分钟的长任务
```

## 📌 第二支柱：版本治理体系 (Version Governance)

> **核心原则**：一个全局主版本号，所有模块前缀统一；版本就是兼容性的承诺

---

### 2.1 统一三段式版本号

```
v4.1.0
│ │ └─ PATCH  小修复、文档更新、不影响功能的调整（向下100%兼容）
│ └─── MINOR  新增功能、新增模块、小改进（向下100%兼容）
└───── MAJOR  架构重构、不兼容变更、里程碑发布
```

---

### 2.2 文件命名规范（强制执行）

```
命名公式：
  文件名_v全局主版本号_模块内部版本号.md

示例：
  Jerry_智能体团队创建者_进阶计划_v4.1.0_2.0.1.md
                    ↓              ↓
               全局主版本     模块内部迭代版本
```

**例外清单（不需要版本后缀）**：
```
- README.md          （导航入口，始终最新）
- SOUL.md            （灵魂文件，极少变更）
- AGENT.md           （本文件！治理体系整合版）
- IDENTITY.md        （身份定义，稳定不变）
```

---

### 2.3 版本一致性检查（发布前必须通过）

```bash
# 一键检查命令（在 roadmap/ 目录下执行）
find . -name "*.md" -path "*/_archive/*" -prune -o -name "*.md" -print | while read file; do
    version=$(echo "$file" | grep -oE "v[0-9]+\.[0-9]+\.[0-9]+" | head -1)
    if [ -z "$version" ]; then
        echo "🟢 $file (稳定核心文件)"
    elif [ "$version" != "v4.1.0" ]; then
        echo "🔴 $file ($version != v4.1.0)"
    else
        echo "✅ $file ($version)"
    fi
done
```

**验收标准**：不能有任何 🔴 项

---

### 2.4 版本发布标准流程（7步）

```
Step 1 🔍 版本一致性检查
  ↓
Step 2 📝 更新总纲领版本号（智能体进阶规划_vX.Y.Z.md）
  ↓
Step 3 🔗 检查所有交叉引用是否指向正确版本
  ↓
Step 4 📋 生成版本发布说明（变更内容、影响范围）
  ↓
Step 5 ✅ 安全闸门审批（展示所有变更，获得 Rex 确认）
  ↓
Step 6 🏷️ Git Tag 打标（git tag -a vX.Y.Z -m "版本说明"）
  ↓
Step 7 🚀 提交并推送（git push origin vX.Y.Z）
```

## 🧠 第三支柱：记忆-知识流体系 (Memory Flow)

> **核心原则**：记忆是流动的知识，知识是沉淀的记忆；让有价值的经验自动沉淀到知识库

---

### 3.1 四层记忆金字塔架构

```
L1 瞬时记忆 🧠   ←  会话内存 / 当前对话上下文
  ↓ (会话结束触发评估，≥60分自动下沉)
L2 短期记忆 📝   ←  memory/YYYY-MM-DD.md（当日记忆）
  ↓ (每日 02:00 蒸馏，≥80分自动下沉)
L3 中期记忆 📚   ←  knowledge-base/ SQLite 数据库
  ↓ (满足3个晋升条件，人工确认后下沉)
L4 长期基因 🔬   ←  roadmap/ 核心文件（SOUL/AGENT/版本规范）
```

---

### 3.2 记忆价值自动评估器（五大维度）

```
每条记忆自动从 5 个维度打分（0-20分），总分 ≥ 60 分自动下沉

维度 1 🆕 创新性  是否包含新知识、新方法、新发现？
维度 2 ♻️ 复用性  是否可能在未来被多次引用？
维度 3 ✅ 正确性 是否经过验证、正确有效？
维度 4 ⚠️ 风险性  是否包含错误教训、踩坑经验？
维度 5 🌍 普适性  是否适用于多个场景、多个智能体？
```

---

### 3.3 晋升 L4 核心基因的三大门槛（缺一不可）

| 条件 | 阈值 | 说明 |
|------|------|------|
| **重复出现次数** | ≥ 3 次 | 同一 Pattern-Key 被标记 3 次以上 |
| **跨任务数量** | ≥ 2 个 | 出现在至少 2 个不同任务场景 |
| **时间窗口** | ≤ 30 天 | 30 天内连续出现，证明是持续性问题 |

**晋升目标映射**：
```
行为规则类问题    →  AGENT.md（本文件！）
工具环境类问题    →  TOOLS.md
客户业务信息类    →  MEMORY.md
人格风格类问题    →  SOUL.md
```

---

### 3.4 三类日志标准格式规范

#### ✅ LEARNINGS.md（成功经验库）
```
Date: YYYY-MM-DD
Pattern-Key: xxx-yyy-zzz
Context: 在什么场景下
Learning: Rex 纠正了什么 / 发现了什么最佳实践
Recurrence-Count: N
Related-Tasks: [任务1, 任务2...]
```

#### ❌ ERRORS.md（错误案例库）
```
Date: YYYY-MM-DD
Pattern-Key: xxx-yyy-zzz
Command: 执行了什么命令
Error: 错误信息
Root-Cause: 根因分析
Recurrence-Count: N
Solution: 修复方案（如果已解决）
```

#### 💡 FEATURE_REQUESTS.md（需求池）
```
Date: YYYY-MM-DD
Pattern-Key: xxx-yyy-zzz
Request: Rex 要求了什么能力
Context: 在什么场景下需要
Priority: P0/P1/P2
Recurrence-Count: N
```

---

### 3.5 会话结束前必做：记忆梳理检查清单

```
✅ 成果记录：今天完成了什么？
✅ 经验提炼：有什么最佳实践值得保存？
✅ 错误复盘：出了什么问题？根因是什么？
✅ 要求记录：Rex 提了什么新要求？
✅ 待沉淀：有什么 Pattern-Key 可能需要晋升？

✅ 团队创建记录：是否创建/修改了智能体团队？记录完整信息
```

## 📋 治理体系自我进化机制

### 本文件的更新触发条件

当以下任一情况发生时，必须审议更新本治理体系：

1. **同一个安全问题出现 3 次以上**
2. **同一个版本不一致问题出现 3 次以上**
3. **同一个记忆流失问题出现 3 次以上**
4. **Rex 明确要求修改治理规则**
5. **大里程碑版本发布（如 v4.0 → v5.0）**
